# 中期答辩问题详答

本文用于回答中期答辩中的高频追问。写法遵循一个原则：所有回答都区分清楚 `自己实现了什么`、`调用了什么第三方能力`、`哪些是主链路`、`哪些只是可选能力或后备方案`，避免把工程集成说成“自研模型”，也避免把自己写的业务算法说成“只是调了个库”。

---

## 一、统一回答原则

### 1. 本系统是“自研业务算法 + 第三方基础能力”的混合方案

本课题并不是从零训练视觉模型、聚类模型或视频生成模型，而是在成熟基础组件之上，针对“旅行相册自动整理与回忆生成”这个具体场景，自己完成了：

- 面向旅行场景的时空距离设计与后处理规则
- 端侧视觉结果的结构化 schema 设计与启发式推断逻辑
- 事件故事、章节、微故事和视频文案的分层生成链路
- 本地视频场景编排、时长压缩、配乐规划和原生导出流水线
- 整套移动端、后端、任务调度、数据结构与交互系统

同时，系统也明确依赖了第三方能力：

- 聚类基础算子：`scikit-learn DBSCAN`、可选 `hdbscan`
- 语义向量能力：可选 `sentence-transformers`
- 端侧视觉基础能力：`Google ML Kit`
- 云端 LLM：`OpenAI-compatible / DeepSeek / 通义`
- 视频编码：`AndroidX Media3 Transformer`
- 地图与地理编码：`高德地图服务`

因此，答辩时更准确的表述不是“全部自己从零做”，也不是“只是拼接口”，而是：

> 本系统采用成熟底层能力作为基础，在其上针对旅行回忆场景做了大量业务层、算法层和工程层的自研实现，最终形成了完整可运行系统。

### 2. 所有问题都建议按“主链路 + 自研部分 + 第三方部分”来回答

如果老师追问某个模块是不是自己写的，建议统一按下面的结构回答：

1. 先说主链路现在真实怎么跑
2. 再说其中哪些部分是自己设计和实现的
3. 最后说底层依赖了哪些第三方库、模型或平台

这样既不会虚夸，也不会把自己的工作量说轻。

## 二、问题 1：时空聚合算法是怎么做的，具体细节是什么，能实现什么效果

### 1.1 建议直接回答

本系统的时空聚合不是简单按日期切分，也不是完全黑盒的现成聚类接口，而是采用了“`时空密度聚类 + 时间规则后处理 + 语义辅助合并`”的混合方案。

其中，基础密度聚类部分确实调用了第三方算法库，当前使用的是 `scikit-learn` 的 `DBSCAN`，在环境支持时也会尝试 `HDBSCAN`；但时空距离矩阵的构建、自适应阈值、小范围拍摄惩罚放宽、跨城市拆分、超长事件切分、小事件合并、夜间噪声过滤这些更贴近旅行场景的逻辑，都是我自己实现的。

因此更准确的说法是：

> 我不是从零重新发明 DBSCAN，而是在成熟聚类算法之上，自己实现了一套面向旅行照片场景的时空聚合策略。

### 1.2 具体处理流程

#### 第一步：数据清洗

进入聚类前，系统先清洗异常元数据：

- 如果拍摄时间早于 `2000-01-01`，认为相机时间异常，直接标记为噪声
- 如果拍摄时间晚于当前时间 `1 天` 以上，也视为异常时间
- 如果 GPS 为 `0,0`、越界，或者缺失，则不作为有效空间坐标参与空间距离计算

这一步的目的，是避免脏数据把整个聚类结果带偏。

#### 第二步：构建时空距离矩阵

系统不会直接把“经纬度 + 时间戳”丢给聚类器，而是先自己定义照片之间的时空距离。

做法是：

- 时间阈值不是写死的，而是根据相邻照片时间间隔的 `75 分位数` 自适应估计
- 空间阈值也不是固定值，而是根据照片地理距离分布的 `80 分位数` 自适应估计
- 对每一对照片，分别计算：
  - 时间差归一化 `time_norm`
  - 空间差归一化 `space_norm`
- 最终距离定义为：

```text
combined_distance = sqrt(time_norm^2 + space_norm^2)
```

另外，为了适配旅行中的“同景点慢慢拍”的情况，还加入了一个专门规则：

- 如果两张照片空间距离在 `500 米` 以内，就把时间惩罚减半

这意味着：同一景点里即使拍摄跨越了更长时间，也不容易被错误拆开。

#### 第三步：基础密度聚类

构建完距离矩阵后，系统会执行基础密度聚类：

- 默认使用 `DBSCAN(metric="precomputed")`
- 如果环境里安装了 `HDBSCAN`，并且样本规模合适，则同时跑 `HDBSCAN(metric="precomputed")`
- 最后比较两者结果，优先选择：
  - 聚到更多有效样本的结果
  - 在有效样本数接近时，簇数更合理、不过度碎片化的结果

也就是说，第三方算法负责“从距离矩阵中找密度簇”，但距离矩阵本身和后续修正不是库默认给的，而是本系统自己定义的。

#### 第四步：时间规则后处理

基础聚类结束后，系统还会做一轮专门面向旅行场景的规则修正。主要包括：

1. `短间隔合并`
   - 如果两个事件间隔很短，默认小于 `180 分钟`
   - 就把它们合并成一个连续旅行片段

2. `事件内部大时间缝隙切分`
   - 如果同一事件内部相邻照片时间差过大，就从中间切开
   - 当 GPS 发生明显跳跃时，也会触发拆分

3. `超长事件切分`
   - 如果一个事件跨度过长，例如持续很多天或照片过多
   - 系统会寻找最合理的切分点，避免整趟旅行被错误合并成一个超大事件

4. `跨城市跳跃拆分`
   - 如果相邻事件的空间中心相距很远，例如超过 `50km` 就更敏感地拆分
   - 如果出现超过 `200km` 的大跳跃，则直接拆开

5. `小事件合并`
   - 如果某个事件照片太少，例如只有 1-2 张
   - 且它和前后事件在时间、空间上都接近，就尝试并入相邻事件

6. `夜间单张噪声过滤`
   - 如果一个事件只有 1 张照片，且时间处于 `22:00-06:00`
   - 系统会把它当作更可能的误触或噪声，不纳入最终事件

这些规则都不是第三方库提供的，而是我针对旅行照片场景自己写的业务层算法。

#### 第五步：语义辅助合并

在基础聚类和时间规则之后，系统还会做一层可选的语义辅助合并。

做法是：

- 从每个事件中挑代表照片
- 为代表照片构造文本上下文：`简要描述 + 拍摄时间 + 位置`
- 如果本地可用，会使用 `sentence-transformers` 的 `clip-ViT-B-32-multilingual-v1` 生成向量
- 如果该模型不可用，则退化为 `HashingVectorizer` 的文本向量方案
- 计算相邻事件之间的向量相似度
- 只有当两个事件在 `语义上相似` 且 `时空上相邻` 时，才会进一步合并

这里也要实事求是地说：

- `SentenceTransformer / CLIP` 是第三方向量模型能力
- 但“挑哪些代表图、如何构造上下文、何时允许合并、如何与时空条件联动”是我自己写的

### 1.3 最终能达到什么效果

这套算法主要解决的是旅行照片天然存在的三个问题：

1. 照片很多，但用户记忆其实是按“事件片段”组织的
2. 同一景点可能拍很久，不能因为时间拉长就乱拆
3. 跨城市、跨天、跨主题活动又不能被错误合并

因此最终效果是：

- 能把一堆离散照片聚成若干“旅行事件”
- 同一景点内的连续拍摄更容易聚在一起
- 不同城市、不同景点之间更容易被拆开
- 超长旅行不会被粗暴合并成一个大事件
- 很小的噪声事件会被过滤或并入相邻事件

### 1.4 自研部分和第三方部分怎么说

| 类型 | 内容 |
| --- | --- |
| 第三方依赖 | `scikit-learn DBSCAN`、可选 `hdbscan`、可选 `sentence-transformers` |
| 自己实现 | 时空距离定义、自适应阈值、小范围时间惩罚放宽、规则后处理、语义辅助合并策略、事件创建与噪声标记 |

### 1.5 不建议说的表述

不建议说：

- “这个聚类算法完全是我原创发明的”
- “我们就是简单调了个 DBSCAN”

更建议说：

> 底层密度聚类算子使用了成熟第三方库，但整个旅行场景下的时空距离设计、规则修正和语义辅助策略是我自己实现的。

### 1.6 对应实现位置

- `backend/app/services/clustering_service.py`

## 三、问题 2：端侧视觉模型使用的什么模型、技术，有什么效果，具体作用和细节是什么，会给云端什么数据

### 2.1 建议直接回答

当前系统的端侧视觉分析不是自己训练的大模型，而是基于 Android 端的 `Google ML Kit` 组合能力来做的轻量本地分析。底层调用了：

- `Face Detection`
- `Image Labeling`
- `Text Recognition`

在这些基础能力之上，我自己实现了面向旅行照片的结构化推断逻辑，把原始识别结果整理成统一的视觉 schema，并回写给后端。

因此更准确的说法是：

> 端侧视觉不是自训练模型，而是基于 Google ML Kit 的本地视觉流水线；我负责的是结果 schema 设计、特征推断、缓存同步以及和后端故事链路的衔接。

这里还可以额外补一句，避免和云端视觉接口混淆：

> 后端代码里虽然也预留了云端视觉分析封装，但当前默认主链路并不是把每张照片上传给 OpenAI 或通义做识图，而是优先采用端侧 ML Kit 先提取结构化结果，再回写后端。

### 2.2 底层用了什么第三方能力

当前 Android 端原生模块 `TravelVision` 直接依赖了以下三方能力：

- `com.google.mlkit:face-detection:16.1.7`
- `com.google.mlkit:image-labeling:17.0.9`
- `com.google.mlkit:text-recognition:16.0.1`

这些能力负责最底层的：

- 检测是否有人脸、人脸数量
- 提取图像标签，例如海边、建筑、食物、交通工具等
- 识别图片中的文字，例如路牌、菜单、地标词

### 2.3 我自己实现了哪些部分

虽然底层检测用了现成组件，但系统真正用于业务的视觉结果并不是直接照搬 ML Kit 原始输出，而是我自己定义并生成了一份结构化结果，主要字段包括：

- `scene_category`
- `object_tags`
- `activity_hint`
- `people_present`
- `people_count_bucket`
- `emotion_hint`
- `ocr_text`
- `landmark_hint`
- `image_quality_flags`
- `cover_score`
- `confidence_map`

其中很多字段都不是 ML Kit 直接返回的，而是我自己通过启发式规则推断出来的。

例如：

#### 1. 场景分类 `scene_category`

系统会把图像标签归一后，再根据关键词映射成更适合旅行故事使用的场景类别，例如：

- `beach`
- `mountain`
- `city`
- `food_and_dining`
- `landmark`
- `transport`
- `indoor`
- `nature`

#### 2. 活动推断 `activity_hint`

活动标签不是直接检测出来的，而是根据：

- 场景类别
- 标签词
- 人脸数量
- OCR 文本

综合推断的，例如：

- `group_photo`
- `beach_walk`
- `dining`
- `outdoor_exploring`
- `in_transit`
- `sightseeing`

#### 3. 地标提示 `landmark_hint`

如果标签或 OCR 中出现桥、塔、教堂、寺庙、博物馆等线索，就推断成地标类别，供后续故事生成使用。

#### 4. 情绪提示 `emotion_hint`

情绪并不是大模型直接生成的，而是结合人脸笑容概率等信息做轻量推断：

- 笑容高，偏 `joyful`
- 中等，偏 `pleasant`
- 否则偏 `neutral`

因此它是启发式情绪标签，不是强语义理解型情绪模型。

#### 5. 图片质量和封面分数

系统还会自己分析图片亮度和对比度，生成：

- `too_dark`
- `too_bright`
- `low_resolution`
- `low_contrast`

同时综合以下因素生成 `cover_score`：

- 图像标签置信度
- 分辨率
- 曝光是否合适
- 是否有人脸、人脸数量
- 是否存在质量问题

这个分数主要用于封面候选和展示质量判断。

### 2.4 端侧视觉的实际作用

端侧视觉结果的主要用途有 4 个：

1. 为后端故事生成提供结构化视觉线索
2. 为照片生成 caption 和 micro story 提供种子信息
3. 辅助封面图选择
4. 让系统在“默认不上原图”的前提下，仍然具备一定视觉语义理解能力

这里特别适合强调课题当前的隐私策略：

> 主链路不是把所有原图上传云端去做识别，而是先在本地提取轻量结构化结果，再把结构化信息回写到后端。

### 2.5 会给云端什么数据

端侧最终回写给后端的不是原始标签数组，也不是图片本身，而是一份结构化 JSON，再附带状态字段。

核心内容包括：

```json
{
  "schema_version": "single-device-vision/v1",
  "source_platform": "android-mlkit",
  "generated_at": "...",
  "scene_category": "...",
  "object_tags": ["..."],
  "activity_hint": "...",
  "people_present": true,
  "people_count_bucket": "2-3",
  "emotion_hint": "...",
  "ocr_text": "...",
  "landmark_hint": "...",
  "image_quality_flags": ["..."],
  "cover_score": 0.82,
  "confidence_map": {
    "scene_category": 0.88,
    "cover_score": 0.82
  }
}
```

同时还会回写：

- `visionStatus`
- `visionError`

后端把这份结果存进 `photos.vision_result(JSON)`，并把其中一部分摘要化为 `visual_desc`、`emotion_tag` 等字段，继续用于故事链路。

### 2.6 效果怎么描述更合适

不建议把它描述成“端侧多模态大模型”。

更准确的描述是：

> 当前端侧视觉模块是一个轻量、可本地运行的视觉分析流水线，能够提取与旅行场景相关的结构化线索，例如场景类别、活动提示、人物数量、OCR 文本、地标提示和封面评分，用于支撑后续故事生成和视频编排。

### 2.7 自研部分和第三方部分怎么说

| 类型 | 内容 |
| --- | --- |
| 第三方依赖 | `Google ML Kit Face Detection / Image Labeling / Text Recognition` |
| 自己实现 | 原生模块封装、schema 设计、场景/活动/地标/情绪/封面分数推断、端侧队列、缓存、回写后端 |

### 2.8 对应实现位置

- `mobile/modules/travel-vision/android/src/main/java/expo/modules/travelvision/TravelVisionModule.kt`
- `mobile/src/services/vision/onDeviceVisionService.ts`
- `mobile/src/services/vision/onDeviceVisionQueueService.ts`

## 四、问题 3：针对视频生产和故事生成，对应的策略是什么

这个问题建议拆成两半回答：`故事生成策略` 和 `视频生产策略`。因为这两个模块虽然相关，但不在同一端完成，也不是同一种技术路线。

### 3.1 故事生成策略

#### 建议直接回答

故事生成采用的是“`结构化线索先行 + 分层生成 + 异步刷新`”的策略，而不是把全部照片直接丢给大模型一次性写长文。

主链路是：

1. 端侧先产出轻量视觉结果
2. 后端把多张照片汇总成事件级结构化线索
3. 再生成事件总故事
4. 再按章节生成章节故事
5. 再按照片组生成单图微故事

这样做的原因，是为了提高生成结果的稳定性、可控性和可解释性。

#### 详细做法

##### 1. 信号清洗与聚合

后端不会直接把 `vision_result` 原样塞进提示词，而是先做统一清洗，包括：

- 去掉坐标串、无意义英文碎片、噪声 OCR
- 过滤低置信度字段
- 把多张图的视觉结果汇总成：
  - `photo_descriptions`
  - `timeline_clues`
  - `structured_summary`
  - `dominant_emotion`

这一步是自己实现的信号整理层，不是大模型自动完成的。

##### 2. 事件级故事生成

事件级故事会调用云端 LLM 生成：

- `title`
- `full_story`
- `hero_title`
- `hero_summary`
- `emotion`

当前后端做了 AI Provider 抽象，支持：

- `OpenAI-compatible`
- `DeepSeek`
- `Tongyi`

当前默认配置是：

- `openai_story_model = gpt-5.1-codex`

如果切到通义，也会使用：

- `qwen-plus` 生成故事

因此这里也需要说清楚：

- `大模型能力` 是第三方
- `提示词设计、信号组织、字段约束、版本刷新逻辑` 是自己实现的

##### 3. 章节生成

系统会把一个事件按照片顺序每 `10 张` 左右切为一章，再为每章生成：

- `chapter_title`
- `chapter_story`
- `slideshow_caption`

这样做的好处是，故事不只是一个总段落，而是有“章节感”，便于详情页展示和视频场景编排。

##### 4. 微故事生成

章节内部又会把照片切成若干小组，每组大约 `2-5 张`，再给每张图生成 15-25 字的 `micro_story`。

这些短文案优先用于视频字幕，效果比直接重复整段故事更自然。

##### 5. 版本控制和异步刷新

故事生成不是一次性静态结果。系统为事件维护：

- `event_version`
- `story_generated_from_version`
- `story_requested_for_version`
- `story_freshness`
- `slideshow_freshness`

只要用户改了事件标题、地点、照片归属或结构，系统就会把故事标记为 `stale`，后台重新生成。

这也是自己实现的重要工程点，不是第三方模型自带的机制。

### 3.2 视频生产策略

#### 建议直接回答

当前视频生产不是“文生视频模型”，而是“`AI 生成文案 + 规则化场景编排 + 本地原生合成`”。

这样做的原因有三个：

1. 与当前系统“默认不上图”的隐私策略一致
2. 工程可控，输出更稳定
3. 更适合毕业设计当前阶段的可运行系统目标

#### 具体做法

##### 1. 场景编排

系统先把事件故事转成视频场景，当前主要有 3 类：

- `photo-frame`：单图场景
- `montage-frame`：拼贴场景
- `title-plate`：章节标题页 / 总结页

字幕优先级大致是：

1. `micro_story`
2. `caption`
3. `slideshow_caption`

##### 2. 时长控制

每个场景时长不是写死的，而是根据文案阅读长度估算。整段视频还会做总时长压缩，当前默认上限约为 `120 秒`。

如果总时长超限，系统会：

- 先压缩单图场景时长
- 再压缩辅助场景时长
- 必要时删除中间一部分照片场景

##### 3. 比例和版式

系统支持：

- `16:9`
- `9:16`

如果用户不手动指定，就根据整组照片横竖占比自动决定更合适的方向。

##### 4. 配乐规划

音乐不是自己生成，也不是训练音乐模型，而是从整理好的 Pixabay 曲库清单中做匹配。

匹配时会综合使用：

- 事件情绪
- 故事文本
- 章节标题和摘要
- 照片视觉标签
- 场景类别
- 时间段

最终输出一个音频片段计划 `audioSegments`，再和视频一起本地合成。

##### 5. 本地原生合成

前端把场景、布局、字幕、音频计划整理成 JSON 后，交给 Android 原生模块本地渲染和编码。

这部分的细节在问题 4 里再展开。

### 3.3 自研部分和第三方部分怎么说

| 模块 | 第三方依赖 | 自己实现 |
| --- | --- | --- |
| 故事生成 | OpenAI-compatible / DeepSeek / 通义 | 信号清洗、提示词结构、分层生成、版本刷新、章节与微故事组织 |
| 视频生产 | Pixabay 音乐素材、Android 原生编码能力 | 场景编排、字幕优先级、时长压缩、比例决策、配乐匹配规则、本地导出流水线 |

### 3.4 不建议说的表述

不建议说：

- “我们做了视频生成大模型”
- “我们完成了原创音乐生成”
- “故事是直接根据所有图片自动写出来的”

更准确的说法是：

> 当前系统采用结构化线索驱动的故事生成，以及规则编排加本地原生合成的视频生产方案。音乐部分是配乐匹配，不是原创音乐模型。

### 3.5 对应实现位置

- `backend/app/services/event_ai_service.py`
- `backend/app/services/chapter_ai_service.py`

## 五、问题 4：生产视频用的是什么技术栈，当前本地视频合成的具体技术细节是什么

### 4.1 建议直接回答

当前视频不是在云端合成，而是在 Android 端本地完成。技术上采用的是“`React Native 负责编排 + Kotlin 原生模块负责渲染和编码`”的方案。

具体来说：

- 前端先把故事、章节、照片和配乐规划成视频时间线
- 再把这份时间线配置传给本地原生模块
- 原生模块逐场景渲染位图
- 最后用 `AndroidX Media3 Transformer` 把静态图片序列和音频片段编码成 `MP4(H.264 + AAC)`

### 4.2 技术栈组成

#### 前端与编排层

- `Expo 54`
- `React Native 0.81`
- `TypeScript`
- `Expo Router`

#### 本地资源处理

- `expo-file-system`
- `expo-image-manipulator`
- `expo-media-library`

#### 原生导出层

- 自定义 Expo 原生模块：`TravelSlideshowExport`
- Android 实现语言：`Kotlin`

#### 视频编码与合成

- `androidx.media3:media3-transformer:1.8.0`
- `androidx.media3:media3-effect:1.8.0`
- `androidx.media3:media3-common:1.8.0`

### 4.3 具体合成方式

#### 第一步：前端生成导出配置

前端会先生成一份 `NativeExportConfig`，里面包含：

- 事件标题
- 导出比例
- 画布尺寸
- 布局参数
- 场景列表
- 字幕列表
- 音频片段列表
- 帧率、码率、关键帧间隔等编码参数

也就是说，前端本质上做的是“视频 DSL/配置层”，把一个事件翻译成一份本地可执行的视频描述。

#### 第二步：本地整理素材

在真正导出前，前端会先做本地预处理：

- 根据导出比例压缩和缩放图片
- 把需要的照片复制到 staging 目录
- 下载或命中本地缓存的音频资源
- 生成导出时间线

当前系统区分 `preview` 和 `export` 两种模式：

- 预览更轻，分辨率和帧率更低
- 导出更高，质量更好

当前主要参数大致是：

| 模式 | 分辨率 | 帧率 | 视频码率 |
| --- | --- | --- | --- |
| 预览 | `960x540` 或 `540x960` | `6 fps` | `1.8 Mbps` |
| 导出 | `1920x1080` 或 `1080x1920` | `12 fps` | `5 Mbps` |

这里也要实事求是说明：当前是“静态场景级渲染”，不是 30fps 的复杂运动视频引擎，也不是生成式视频模型。

#### 第三步：原生逐场景渲染位图

进入原生模块后，会为每个场景生成一张 Bitmap，再保存为临时 JPEG。

当前支持 3 类场景：

1. `photo-frame`
   - 单张照片铺满画面
   - 使用 `fitCenter`

2. `montage-frame`
   - 最多选 3 张图做拼贴
   - 使用 `centerCrop`
   - 支持圆角拼贴块

3. `title-plate`
   - 可以先铺背景图
   - 叠加半透明黑色遮罩
   - 再绘制标题、章节标题和正文

文本绘制不是网页截图，而是原生 `Canvas + StaticLayout + TextPaint` 手工排版，所以字体、行高、居中和多行截断都在本地控制。

#### 第四步：音视频合成

原生模块会把所有场景临时图变成一个图片序列，每张图通过：

- `MediaItem.setImageDurationMs(...)`

指定显示时长。

音频部分会把每个音乐片段裁成：

- `sourceStartMs`
- `sourceEndMs`

再作为音频轨拼进同一个 `Composition`。

最后由 `Media3 Transformer` 执行导出，使用：

- 视频编码：`H.264`
- 音频编码：`AAC`

#### 第五步：导出完成后的本地保存

导出完成后，系统会：

- 删除临时渲染目录
- 把 MP4 保存到本地缓存目录
- 在“正式导出”模式下再写入系统相册并支持分享

### 4.4 这个方案的特点

优点：

- 完全本地，不依赖云端视频渲染
- 符合当前“默认不上图”的隐私策略
- 工程路径稳定，可控、可演示、可复现

限制：

- 当前更偏“高质量静态幻灯片视频”，不是复杂 GPU 特效视频编辑器
- 当前主要支持 Android 端原生导出
- 当前没有引入 FFmpeg，也没有自研视频生成模型

### 4.5 自研部分和第三方部分怎么说

| 类型 | 内容 |
| --- | --- |
| 第三方依赖 | Expo 生态、AndroidX Media3 Transformer、Android 原生 Canvas/Bitmap API |
| 自己实现 | 时间线编排、布局合同、场景渲染逻辑、字幕组织、素材预处理、预览/导出双模式、本地导出流水线 |

### 4.6 不建议说的表述

不建议说：

- “我们用了 AI 视频生成模型”
- “我们在云端做了视频渲染”
- “我们自己实现了底层视频编码器”

更准确的说法是：

> 当前系统采用本地原生渲染和编码方案，把故事文案、照片、字幕和配乐规划转成可导出的旅行回忆视频。

### 4.7 对应实现位置

- `mobile/src/services/slideshow/slideshowExportService.ts`
- `mobile/src/services/slideshow/slideshowSceneBuilder.ts`
- `mobile/src/services/slideshow/slideshowVideoContract.ts`
- `mobile/src/services/slideshow/slideshowAudioService.ts`
- `mobile/modules/travel-slideshow-export/android/src/main/java/expo/modules/travelslideshowexport/TravelSlideshowExportModule.kt`

## 六、问题 5：用的什么数据库，大概的数据库表关系是怎样的

### 5.1 建议直接回答

当前后端使用的是关系型数据库，开发默认是 `SQLite`，同时兼容 `PostgreSQL`。ORM 采用 `SQLAlchemy`，数据库迁移采用 `Alembic`。

数据库设计并不是通用相册产品的大而全 schema，而是围绕“旅行事件、故事生成、视频回忆”这条主链路来组织。

### 5.2 当前数据库技术栈

- 数据库：`SQLite` 默认，兼容 `PostgreSQL`
- ORM：`SQLAlchemy 2.x`
- 迁移：`Alembic`
- 后端框架：`FastAPI`

这里也需要区分一下：

- 云端结构化数据使用数据库
- 手机本地并没有单独搞一套复杂 SQLite 业务库
- 当前移动端主要使用的是 `AsyncStorage + FileSystem` 做本地缓存和媒体映射

所以如果老师问“数据库”，默认回答的是后端数据库。

### 5.3 核心表关系

#### 1. `users`

用户表，当前主要支撑：

- `device_id`
- `email`
- `nickname`
- `avatar_url`

它与以下表是一对多关系：

- `users -> photos`
- `users -> events`

#### 2. `photos`

这是照片结构化信息表，保存的是照片 metadata 和分析结果，而不是完整原图本体。

核心字段包括：

- `asset_id`
- `gps_lat / gps_lon`
- `shoot_time`
- `width / height / file_size`
- `event_id`
- `status`
- `caption`
- `micro_story`
- `vision_result(JSON)`
- `vision_status`
- `vision_error`

关系上：

- 多张 `photos` 属于一个 `event`

#### 3. `events`

这是系统最核心的业务表，表示一段旅行事件。

核心字段包括：

- `title`
- `location_name`
- `gps_lat / gps_lon`
- `start_time / end_time`
- `photo_count`
- `cover_photo_id`
- `story_text / full_story`
- `hero_title / hero_summary`
- `emotion_tag`
- `status`
- `event_version`
- `story_generated_from_version`
- `story_requested_for_version`
- `story_freshness`
- `slideshow_freshness`

它不仅保存事件内容，还承担了“故事和视频版本状态管理”的职责。

#### 4. `event_chapters`

每个事件会按照片顺序切成若干章节。

主要保存：

- `chapter_index`
- `chapter_title`
- `chapter_story`
- `chapter_intro`
- `chapter_summary`
- `slideshow_caption`
- `photo_start_index / photo_end_index`

关系上：

- 一个 `event` 对应多个 `event_chapters`

#### 5. `photo_groups`

章节内部还会把照片切成更小的组，用于微故事和段落组织。

主要保存：

- `group_index`
- `group_theme`
- `group_emotion`
- `group_scene_desc`
- `photo_start_index / photo_end_index`

关系上：

- 一个 `chapter` 对应多个 `photo_groups`

#### 6. `async_tasks`

这个表用于记录后台异步任务，例如：

- 聚类任务
- 地理编码任务
- AI 故事生成任务
- 增强故事生成任务

主要保存：

- `task_type`
- `status`
- `stage`
- `progress`
- `result`
- `error`

这个表对答辩也很重要，因为它说明系统不是纯算法脚本，而是有完整的后台任务编排机制。

#### 7. `event_enhancement_assets`

这是增强故事生成时使用的代表图临时存储表。

主要保存：

- 属于哪个事件
- 来自哪张照片
- 本地路径 / 公网 URL
- 文件大小
- 过期时间

它说明系统支持“用户主动上传少量代表图做增强故事”这条可选链路，但这不是默认主链路。

### 5.4 一个简化的表关系描述

可以在答辩时这样口头描述：

```text
User
 ├── Photo (多张)
 ├── Event (多个)

Event
 ├── Photo (多张)
 ├── EventChapter (多章)
 ├── EventEnhancementAsset (可选增强图)

EventChapter
 ├── PhotoGroup (多组)
```

如果要再补一句，可以说：

> 其中 `photos` 存原始结构化元数据和端侧视觉结果，`events` 存聚合后的事件和故事结果，`event_chapters/photo_groups` 则负责把事件进一步组织成可展示、可播放的叙事结构。

### 5.5 这个数据库设计的特点

1. 它不是单纯的“照片表 + 用户表”
2. 它已经把“事件聚合、故事生成、视频展示”三层结构落进数据库
3. 视觉结果没有单独拆很多表，而是保存在 `photos.vision_result(JSON)` 中，方便快速演进
4. 故事和视频采用版本号与 freshness 字段管理，适合异步重生成

### 5.6 自研部分和第三方部分怎么说

| 类型 | 内容 |
| --- | --- |
| 第三方依赖 | `SQLite / PostgreSQL`、`SQLAlchemy`、`Alembic` |
| 自己实现 | 面向旅行事件的表结构设计、字段语义、版本管理、事件-章节-照片组叙事关系 |

### 5.7 不建议说的表述

不建议说：

- “数据库就是简单存照片”
- “系统只有 SQLite，没有别的设计”

更准确的说法是：

> 数据库层除了存照片 metadata，还承载了事件聚合结果、故事版本状态、章节结构、微故事组织和异步任务进度，是整个系统结构化能力的核心。

- `backend/app/core/config.py`

- `backend/app/models/user.py`
- `backend/app/models/photo.py`

## 八、当前文档对应的真实实现口径

这份文档严格按当前代码实现整理，当前真实口径包括：

- 时空聚合是“自研策略 + 第三方聚类算子”，不是纯原创聚类模型

- 音乐部分是“配乐匹配”，不是原创音乐生成

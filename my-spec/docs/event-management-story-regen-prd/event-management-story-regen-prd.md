# 需求文档：事件管理与故事重生成机制

- 文档状态：Draft
- 优先级：P0
- 负责人：AI 协作
- 创建时间：2026-04-01
- 最后更新：2026-04-01

## 1. 背景

- 当前现状：
  - 当前默认链路已经转向“最近 200 张 -> metadata 去重 -> 自动聚合事件 -> 端侧结构化结果 -> 云端故事”
  - 但产品仍缺少完整的事件管理能力，用户目前主要只能更换事件封面，无法顺畅修正聚类误判
  - 现有系统虽然已有部分事件创建、更新、删除和照片改归属的底层接口，但前端没有形成完整入口，也缺少统一的故事重生成规则
  - 事件一旦允许编辑，就会直接影响故事文本、章节和幻灯片内容，当前缺少明确的状态口径与任务触发机制
- 为什么要改：
  - 自动聚类只能做到“大多数合理”，无法保证所有照片都完全归属正确
  - 如果不给用户纠偏手段，聚类质量只要有少量误判，就会放大为故事和幻灯片质量问题
  - 如果给了编辑能力但没有重生成规则，就会出现“事件结构已经变了，但故事仍是旧版本”的不一致
- 触发这次变更的原因：
  - 已经明确产品方向是“自动聚类为主，人工纠偏为辅”
  - 已经确认事件编辑会成为主链路的一部分，而不是边缘后台功能
  - 当前需要把“事件编辑行为”和“故事/幻灯片派生结果刷新机制”正式定义清楚

## 2. 目标

- 本次要解决什么问题：
  - 为事件补齐完整的增删改能力，包括事件标题/基础信息编辑与事件内照片归属调整
  - 明确哪些用户行为会导致故事重新生成，哪些不会
  - 明确故事何时触发生成，以及当端侧识别仍在异步进行时如何处理
  - 避免“旧任务晚返回覆盖新结果”的版本错乱问题
  - 为后续幻灯片刷新机制定义一致的派生结果口径
- 预期结果是什么：
  - 自动聚类先给出一版可用事件，用户只在必要时纠偏
  - 当事件照片集合发生变化时，系统可自动把受影响事件重新纳入故事生成流程
  - 故事生成基于当前事件的最新照片集合，而不是过期版本
  - 用户不会因为连续编辑而触发无意义的多次大模型请求
- 成功标准是什么：
  - 用户可对事件执行基础增删改和照片归属调整
  - 事件照片集合变更后，故事状态可被正确标记为“待更新”
  - 仅当该事件当前照片集合的端侧识别达到可生成条件时，才触发故事生成
  - 旧版本生成结果不会覆盖新版本结果

## 3. 非目标

- 这次明确不做什么：
  - 不在 P0 做复杂的事件拆分向导或事件合并助手
  - 不在 P0 做照片拖拽排序与手动时间线编排
  - 不在 P0 做章节级或幻灯片逐页人工编辑器
  - 不在 P0 做“照片识别任务取消”能力
  - 不在这次需求中重做端侧识别模型或故事 Prompt 本身
- 哪些相关问题先不处理：
  - 不在 P0 做事件编辑历史记录和撤销/重做
  - 不在 P0 做照片级批量暂停/恢复识别
  - 不在 P0 做多端并发编辑冲突处理

## 4. 用户场景

- 用户是谁：
  - 默认依赖自动聚类，但希望在少量误判时快速修正的普通用户
  - 关注旅行回忆质量，希望故事与事件结构一致的用户
- 典型使用场景：
  - 用户导入后发现某张照片被聚到错误事件，需要把它移出并归到另一个事件
  - 用户发现一个事件标题不准确，希望自己修改事件名称
  - 用户发现一组漏聚的照片，希望从当前事件中选出若干照片新建事件
  - 用户对一个事件做了多次照片调整后，希望系统自动得到基于最终结构的新故事
  - 某个事件有新增照片时，端侧识别仍在后台进行，用户希望系统在识别完成后再生成，而不是提前生成半成品故事
- 关键操作路径：
  - 自动聚类得到事件 -> 用户查看事件详情 -> 修改标题 / 调整照片归属 -> 事件进入待更新状态
  - 若事件仍有照片在端侧识别中 -> 等待该事件达到可生成条件 -> 云端按事件最新版本生成故事
  - 故事生成完成 -> 事件详情和幻灯片消费新版本结果

## 5. 需求拆解

### 5.1 功能需求

- 需求 1：支持事件基础 CRUD
  - 支持创建事件
  - 支持编辑事件标题
  - 支持编辑地点相关字段
  - 支持删除事件
- 需求 2：支持事件内照片归属管理
  - 支持从事件中移除照片
  - 支持把照片移动到另一个已有事件
  - 支持选择若干照片新建事件
  - 支持把无事件照片加入事件
- 需求 3：支持定义“哪些编辑行为触发故事重生成”
- 需求 4：支持把故事状态与“内容是否过期”分开表达
- 需求 5：支持按事件粒度判断“是否可生成故事”
- 需求 6：支持在端侧识别仍未完成时延迟生成，而不是提前生成半成品
- 需求 7：支持同一事件连续多次编辑时合并为最后一次生成，避免重复请求大模型
- 需求 8：支持通过事件版本号防止旧生成结果覆盖新结果
- 需求 9：为幻灯片等派生结果预留相同的“基于哪一版事件生成”机制

### 5.2 交互 / 界面要求

- 页面或入口：
  - 事件详情页需要提供事件编辑入口
  - 照片网格需要提供“移出事件 / 移动到其他事件 / 新建事件”等操作入口
  - 事件列表和详情页都需要展示故事状态与是否待更新
- 用户操作：
  - 用户可修改事件标题
  - 用户可删除事件
  - 用户可选择一个或多个照片调整归属
  - 用户可在事件编辑后不手动点“重新生成”，默认由系统自动处理
- 状态反馈：
  - 事件应能区分：
    - `waiting_for_vision`：事件结构已变更，但仍有照片未完成端侧识别
    - `ai_pending`：照片已满足生成条件，等待生成
    - `ai_processing`：正在生成
    - `generated`：最新故事已完成
    - `ai_failed`：生成失败
  - 除状态外，还应展示故事新鲜度：
    - `fresh`：当前故事对应当前事件版本
    - `stale`：当前故事是旧版本，待更新
  - 当用户编辑事件后，如旧故事仍保留，需明确标记“内容待更新”
- 异常提示：
  - 如果事件当前仍有照片在识别中，提示“照片仍在整理，完成后会自动更新故事”
  - 如果故事生成失败，提示“稍后可自动重试或手动重试”
  - 如果目标事件不存在或照片不属于当前用户，需给出明确错误提示

### 5.3 数据 / 状态要求

- 需要新增或调整的数据：
  - 事件版本号 `event_version`
  - 当前故事所基于的事件版本 `story_generated_from_version`
  - 当前幻灯片所基于的事件版本 `slideshow_generated_from_version`
  - 故事新鲜度 `story_freshness`
  - 事件是否存在待处理结构变化 `has_pending_structure_changes`
  - 照片级端侧识别状态继续保留为 `pending / processing / completed / failed / unsupported`
- 状态流转：
  - 事件结构变化：
    - 用户编辑事件或调整照片集合
    - `event_version + 1`
    - `story_freshness = stale`
    - 若该事件仍有照片处于 `pending / processing`，则事件故事状态为 `waiting_for_vision`
    - 若该事件所有照片均到达终态，则进入 `ai_pending`
  - 事件故事：
    - `waiting_for_vision -> ai_pending -> ai_processing -> generated / ai_failed`
  - 幻灯片：
    - P0 可先沿用故事版本变化规则，只定义状态口径，不强制同步实现完整刷新链路
- 持久化要求：
  - 云端持久化事件版本号、故事版本号、新鲜度状态和任务状态
  - 本地持久化端侧识别队列和必要缓存
  - 默认不持久化原图或缩略图到云端

## 6. 触发规则定义

### 6.1 会触发故事重生成的行为

- 事件新增照片
- 事件移除照片
- 照片从事件 A 移动到事件 B
- 选中照片新建事件
- 删除事件并把照片归入其他事件
- 修改地点相关字段，若这些字段参与故事 Prompt
- 修改会影响照片集合摘要或事件叙事输入的其他结构化字段

### 6.2 默认不触发故事重生成的行为

- 仅更换事件封面
- 仅修改事件标题
- 仅修改用户手填说明且该说明不参与 AI Prompt

### 6.3 标题的特殊规则

- 事件标题默认视为用户可覆盖字段
- 若用户手动修改过标题，后续故事重生成不得再自动改写标题
- 若事件从未被用户手动改标题，系统仍可在首次生成时给出 AI 标题

## 7. 故事生成时机

- 核心原则：
  - 不是“全局所有照片识别完再生成”
  - 而是“某个事件当前照片集合达到可生成条件后，再按事件生成”
- 事件可生成条件：
  - 事件至少有 1 张照片
  - 事件有明确的开始时间和结束时间
  - 该事件当前照片集合中不存在 `pending / processing` 状态
  - 该事件至少有 1 张照片的端侧结构化结果为 `completed`
- 终态定义：
  - `completed`
  - `failed`
  - `unsupported`
- 对失败照片的处理：
  - 失败或不支持的照片不应阻塞整事件永久无法生成
  - 生成时仅使用已完成照片的结构化结果
  - 未完成或失败照片可退化为只使用 metadata，或不参与内容描述聚合

## 8. 版本与去重机制

### 8.1 为什么需要版本

- 用户可能连续多次编辑同一事件
- 端侧识别可能晚于事件编辑完成
- 云端故事任务可能存在并发或晚返回
- 如果没有版本号，旧任务返回结果可能覆盖最新事件结构对应的新故事

### 8.2 版本规则

- 每次事件照片集合发生变化时：
  - `event_version + 1`
- 仅当故事成功基于当前事件版本生成完成时：
  - `story_generated_from_version = event_version`
  - `story_freshness = fresh`
- 若返回结果对应的是旧版本：
  - 不得覆盖当前事件故事内容
  - 任务结果应直接丢弃或仅记录日志

### 8.3 连续编辑合并

- 用户连续多次编辑同一事件时：
  - 不要求每次都立即发起云端故事生成
  - 系统只需确保“最后一个稳定版本”会进入生成
- 推荐策略：
  - 结构变化时先更新版本和状态
  - 若仍在等待端侧识别，则不生成
  - 一旦事件达到可生成条件，则只为当前版本发起一次生成

## 9. 幻灯片联动口径

- 幻灯片应视为和故事并列的派生结果，而不是静态产物
- 事件结构变化后：
  - 幻灯片结果也应视为基于旧版本
  - 至少需要支持把幻灯片标记为“待更新”
- P0 建议：
  - 先把版本机制与状态字段补齐
  - 故事刷新优先自动完成
  - 幻灯片可先沿用“基于故事结果刷新”的既有链路，但必须具备 stale 标记
- 后续若做强化：
  - 可增加 `slideshow_freshness`
  - 可增加 `slideshow_generated_from_version`

## 11. 方案约束

- 必须遵守的业务约束：
  - 自动聚类是主路径，人工纠偏是补救与定稿手段
  - 用户手工修正后的结果优先级高于自动聚类与 AI 生成
  - 默认链路仍不上图，只上传 metadata 与端侧结构化结果
  - 标题等用户 override 字段不得被后续 AI 生成覆盖
- 性能约束：
  - 不应因同一事件的连续多次编辑重复请求大模型
  - 不应因单张失败照片把整事件永久卡住
  - 编辑事件后状态刷新和 UI 反馈应足够快
- 安全 / 隐私约束：
  - 事件编辑和故事重生成不改变默认不上图原则
  - 后台文案不得误导用户认为会上传原图
- 兼容性约束：
  - 不以旧的“事件几乎不可编辑”心智为优先
  - 当前产品方向允许直接删改旧入口和旧行为

## 12. 风险与待确认

- 风险 1：
  - 如果故事 stale 但页面仍显示旧内容，用户可能误以为系统没有生效，需要明确状态提示
- 风险 2：
  - 如果标题既允许用户改，又允许 AI 重算，容易发生覆盖冲突，需要明确“标题是否永久用户优先”
- 风险 3：
  - 若事件版本机制不完整，异步任务晚返回时会覆盖新内容，造成严重数据错乱
- 风险 4：
  - 如果照片移动频繁，章节和幻灯片的刷新策略不清晰，会让用户感知为“故事更新了，但播放内容没更新”
- 还需要确认的问题：
  - 事件标题是否百分之百视为用户覆盖字段
  - 地点字段中哪些参与 Prompt，哪些只用于展示
  - 幻灯片 P0 是只做 stale 标记，还是同步做自动刷新
  - 删除事件时照片是进入“无事件”状态，还是必须要求归入其他事件

## 15. 发布影响

- 是否需要迁移数据：
  - 需要，预计要新增事件版本号、故事版本号、新鲜度等字段
- 是否影响现有用户：
  - 会影响事件详情页的编辑能力、事件状态展示和故事刷新节奏
- 是否需要重新 build / 安装：
  - 需要，前端会新增事件编辑和照片归属调整入口，后端会新增状态与任务编排逻辑
- 回滚方式：
  - 若未上线，可回滚事件版本字段和自动重生成逻辑，恢复到“手动触发重生成或弱状态模型”的旧链路

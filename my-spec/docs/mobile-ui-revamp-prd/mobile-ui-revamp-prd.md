## 1. 背景

- 当前现状：
  - 当前主壳层由 `足迹 / 旅程 / 我的` 三个 Tab 组成，但首屏主入口仍然偏“地图工具”和“状态面板”，没有把“自动整理旅行回忆”做成第一眼心智。
  - `地图页`、`事件列表页`、`事件详情页`、`我的页` 各自都能工作，但视觉语言不统一：卡片层级、按钮样式、图标体系、阴影、渐变、状态提示都存在割裂。
  - 当前配色以暖米色 + 深绿为基础，但落在页面上偏旧、偏闷，缺少现代移动端产品常见的克制感、留白和秩序感。
  - 任务状态通过 Banner、Snackbar、Modal、Chip、空态文案分散出现，用户会感到“App 一直在提示”，但又说不清当前到底处于什么阶段。
  - `我的页` 仍然更像传统设置列表，不像“单设备、隐私优先、本地回忆工具”的信任中心。
- 为什么要改：
  - 当前产品方向已经明确是“单设备、隐私优先、自动整理旅行回忆”，不是地图工具，也不是素材管理工具。
  - 按当前产品方向，`自动导入最近照片` 是主链路，`手动补导入` 是补充入口，界面必须围绕这个主任务重构。
  - 现有 UI 已经成为产品感知短板。即使底层链路正确，界面仍会把产品呈现成“功能堆叠的原型”，而不是成熟移动产品。
- 触发这次变更的原因：
  - 用户明确反馈当前界面“太丑”，并允许彻底重构页面布局、色彩和心智，不保留历史包袱。
  - 本项目当前没有历史兼容负担，可以直接按新的产品表达重做壳层与视觉系统。

## 2. 目标

- 本次要解决什么问题：
  - 把 App 从“若干功能页的集合”重构成“一个会自动整理旅行回忆的移动产品”。
  - 建立统一的移动端设计语言，解决导航、层级、配色、状态反馈、卡片样式割裂的问题。
  - 重新定义首页与核心路径，让用户打开 App 后第一时间看到“最近回忆、整理进度、可继续回看的事件”。
  - 让首页、地图、事件详情、照片相关页面、个人中心和任务中心形成一套统一且精致的产品体验。
- 预期结果是什么：
  - Tab 层级、首页、详情页、地图页、我的页都进入同一套视觉系统和交互规则。
  - 首页改为主入口，地图降级为探索入口，不再承担产品主心智。
  - 首次无数据场景、处理中场景、错误场景、完成场景都有统一且克制的反馈方式。
  - 用户能明显感知到这是“旅行回忆产品”，而不是“相册工具 + 地图页 + 设置页”。
- 成功标准是什么：
  - 冷启动无数据时，用户在 1 次主点击内即可启动“整理最近 200 张”。
  - 有数据时，首屏无需滚动即可看见最近一个回忆卡片和当前整理状态。
  - 事件详情页可在 1 次主点击内进入幻灯片播放。
  - 视觉上不再混用多套按钮、图标、卡片和状态样式。
  - 所有异步任务都能收敛到统一的“整理状态中心”，不再四处分散冒提示。

### 2.2 设计主题

- 新主题名称：`Calm Native Editorial`
- 关键词：
  - 原生感
  - 克制
  - 高级留白
  - 轻编辑感
  - 媒体优先
  - 低噪音状态反馈
- 参考方向：
  - 交互秩序参考高频移动端产品的克制导航与密度控制，尤其是微信这类日活产品的“稳”和“轻”。
  - 媒体浏览和回忆组织参考 Apple Photos / Google Photos 的“回忆优先、照片驱动、自动整理”表达。
  - 最终目标不是模仿任何单一产品，而是形成适合本项目定位的一套独立界面语言。

### 2.3 效果预览

- 首次打开 App：
  - 不是直接看到地图，也不是一堆说明文案，而是一张很干净的全屏欢迎状态页。
  - 页面主体是一张大图感的“回忆启动卡”，下面只有三条简短隐私承诺和一个明确主按钮：`整理最近 200 张`。
  - 整体像一个成熟相册产品的冷启动页，而不是导入弹窗。
- 日常进入 App：
  - 顶部是简洁标题和一个低打扰的“整理中”胶囊。
  - 首屏是一张最近旅程 Hero 卡，直接显示封面、地点、时间和“继续回看 / 播放回忆”。
  - 下方是按月份折叠的回忆流，不再是强时间轴控件，而是更轻、更媒体化的内容卡片。
- 打开事件详情：
  - 顶部是沉浸式封面和简洁操作区，往下是章节卡、故事引子、相册拼贴和完整故事折叠阅读。
  - 页面更像一本会自动排版的旅行记忆册，而不是资料页。
- 打开地图：
  - 地图保持全屏，但控制元素更少，顶部是轻量筛选胶囊，底部是半透明事件抽屉。
  - 地图成为“从空间重新浏览回忆”的入口，而不是应用首页。

## 3. 非目标

- 这次明确不做什么：
  - 不在本期重构后端事件聚类、故事生成、幻灯片导出算法。
  - 不新增社交、分享广场、评论、协作等产品能力。
  - 不做暗色模式第一期适配，本期只把浅色模式打磨到位。
  - 不为了“像某个大厂 App”去逐像素模仿现有产品。
  - 不重构视频播放页 / 幻灯片播放器页面的视觉与策略，本轮默认保持现有播放器设计方案不动。
- 哪些相关问题先不处理：
  - 不在本期引入复杂 3D 动画、粒子特效、沉浸式视频转场。
  - 不在本期新增完整搜索体系和复杂筛选面板。
  - 不在本期重做导出页、系统分享页等低频页面。
  - 不因为本轮视觉升级去改动播放页中的生成策略、导出策略、节奏控制和播放心智。

## 4. 用户场景

- 用户是谁：
  - 有大量旅行照片，希望系统自动整理成事件、故事和回看内容的普通用户。
  - 对图片隐私敏感，希望默认不上传原图的用户。
- 典型使用场景：
  - 用户首次授权相册后，希望一键进入自动整理，而不是自己做素材管理。
  - 用户再次打开 App，希望直接看到最近一次旅行、最近完成的故事、仍在整理中的任务。
  - 用户点进某个事件，希望先浏览照片和片段，再决定是否读完整故事或播放幻灯片。
  - 用户在地图上回看某次旅行的空间轨迹时，希望地图是沉浸式辅助视角，而不是噪音很大的工具页。
  - 用户进入“我的”页时，希望看到设备身份、隐私说明、导入任务、本地数据管理，而不是传统账号中心。
- 关键操作路径：
  - 首次打开 -> 授权相册 -> 整理最近 200 张 -> 查看最近生成的回忆卡片
  - 回忆首页 -> 点击最近事件 -> 浏览章节 -> 播放幻灯片
  - 回忆首页 -> 查看整理中状态 -> 进入导入任务页
  - 地图页 -> 点击聚合点 -> 查看事件卡片 -> 进入详情
  - 我的页 -> 查看隐私策略 / 本机信息 / 清理本地数据

## 5. 需求拆解

### 5.1 功能需求

- 需求 1：重构 App 壳层导航
  - Tab 从当前 `足迹 / 旅程 / 我的` 重构为 `回忆 / 地图 / 我的`
  - `回忆` 成为首页和主入口，承接当前事件列表、首次导入入口、整理状态、最近回忆
  - `地图` 降级为辅助探索入口，但保留完整能力
- 需求 2：重构首次无数据场景
  - 首次无数据时，不再显示零散的空态卡片和重复隐私提示
  - 改为一个完整的冷启动欢迎态，承载：
    - 产品一句话定位
    - 默认不上图的隐私承诺
    - 主动作 `整理最近 200 张`
    - 次级动作 `手动补导入`
- 需求 3：重构回忆首页
  - 首页首屏必须包含：
    - 最近回忆 Hero 卡
    - 统一的整理状态入口
    - 回忆流列表
  - 回忆流支持至少三类筛选：
    - 全部
    - 整理中
    - 待更新
  - 事件列表从“时间轴组件”改为“媒体优先的回忆流”
- 需求 4：重构地图页
  - 地图保持全屏浏览
  - 事件选择改为更轻的底部抽屉式交互
  - 顶部只保留必要操作：返回上一层级、重置视野、筛选状态
  - 地图内状态提示与无定位空态需进入新视觉系统
- 需求 5：重构事件详情页
  - 顶部 Hero 需要更沉浸，但信息更少、更准
  - 主操作收敛为：
    - 播放回忆
    - 查看照片
    - 更多操作
  - `刷新故事` 不再作为详情页主操作
  - 用户编辑事件、调整照片、补导入后，系统默认自动标记待更新并后台刷新
  - 手动重试只作为失败或异常时的弱入口，收纳在 `更多操作`
  - 内容顺序改为：
    - 故事引子
    - 旅程章节
    - 相册
    - 完整故事
  - 章节卡强化媒体拼贴和片段阅读感
- 需求 6：统一任务与状态反馈
  - 导入、端侧分析、故事生成、失败、待更新等状态统一进入“整理状态中心”
  - 常驻状态以内联胶囊、状态卡、页面顶部小条承载
  - Snackbar 只用于短暂结果反馈，不再承担主要状态传达
- 需求 7：重构“我的”页为信任中心
  - 页面重点从“设置列表”改为“这台设备上的旅行回忆”
  - 强化：
    - 本机身份
    - 隐私说明
    - 导入与整理任务入口
    - 本地数据管理
  - 弱化传统设置项堆叠感
- 需求 8：统一幻灯片与照片浏览器外观
  - 本轮不改幻灯片播放器页面本身
  - 照片查看器、封面选择器、底部 Sheet、照片管理、导入选择器要共享统一圆角、按钮、层级和控件样式
  - 需要补齐照片相关真实能力的设计表达：
    - 长按照片操作
    - 批量移动 / 删除
    - 移动目标选择
    - 封面选择 / 恢复默认
    - 权限恢复
    - 导入任务回看

### 5.2 交互 / 界面要求

- 页面或入口：
  - `回忆` 为唯一主入口
  - 首页不再以地图或设置作为第一视图
  - 首次主 CTA 必须固定为 `整理最近 200 张`
- 用户操作：
  - 首页主操作应支持单手完成，主按钮、Hero 卡、事件卡片均需满足 44x44 以上点击热区
  - 事件卡片点击进入详情，长按只保留真正高频的上下文动作
  - 重要操作不依赖 hover，仅依赖明确的点击反馈
- 状态反馈：
  - 加载优先使用骨架屏和内容占位，减少“空白 + 转圈”
  - 后台任务优先以内联状态卡或页内胶囊展示
  - 全局只保留一个统一任务入口，不再同时在多个地方重复提示
- 异常提示：
  - 权限拒绝时使用明确的全页解释 + 跳设置动作
  - 生成失败时优先在事件卡片和详情内提示，不先弹全局打断式对话框
  - 致命错误才使用 Modal / Alert

#### 5.2.1 视觉系统

- 色彩方向：
  - 不再继续使用偏闷的暖米色 + 深绿大面积铺底
  - 新视觉基底改为更清透的雾白、冷白和蓝灰层次，整体更明快、清晰、现代
  - 主强调色以清透蓝为主，暖橙只作为少量情绪和提醒辅助，不再承担大面积主色任务
- 建议色板：

```txt
Background / 雾蓝白: #F3F7FC
Surface / 冷白: #FBFDFF
Surface Strong / 纯白: #FFFFFF
Surface Soft / 浅蓝灰: #EDF3FB
Primary Text / 墨蓝黑: #0F172A
Secondary Text / 石板灰: #5B6678
Divider / 细描边: #D9E3F0
Primary Accent / 清透蓝: #3C67F3
Primary Accent Soft / 轻蓝雾: #E4ECFF
Warm Accent / 暖橙: #FF8C68
Warm Accent Soft / 暖雾橙: #FFE8DF
Map Accent / 地图蓝: #4B82FF
Success / 清稳绿: #2E8B6F
Warning / 琥珀: #B77931
Danger / 赤陶红: #C15D4D
```

- 字体策略：
  - 第一阶段优先使用系统中文字体栈，保证原生感和性能稳定
  - 标题通过字号、字重、留白建立高级感，不依赖花哨字体
  - 若后续需要提升“故事感”，只在事件大标题或章节标题局部引入 `Noto Serif SC`
- 图标策略：
  - 全应用统一单一图标体系，不再混用不同风格的图标集
  - 禁止使用 emoji 充当 UI 图标
- 卡片策略：
  - 主要卡片圆角统一为 `20-24`
  - 卡片采用“细边框 + 极轻阴影”而非厚重投影
  - 一屏内卡片层级不超过三层

#### 5.2.2 页面级视觉要求

- 回忆首页：
  - 顶部采用简洁大标题 + 状态胶囊
  - Hero 卡优先展示最近旅程的大图和一句摘要
  - 回忆流卡片采用大图 + 精简元信息，不再使用强时间轴引导线
- 地图页：
  - 地图层尽量纯净
  - 所有说明和状态优先收纳到顶部浮层或底部抽屉
  - Marker 与底部事件卡片视觉语言一致
- 事件详情：
  - 顶部封面全宽沉浸
  - 操作区需要明显主次，主按钮固定给 `播放回忆`
  - `查看照片` 和 `更多` 为次级入口，不得与主按钮抢权重
  - 自动更新状态使用轻状态条表达，避免首屏出现过长说明
  - 章节卡要有“片段感”，像一本回忆册中的分页，而不是普通折叠卡片
- 我的页：
  - 首屏不是设置分组，而是设备身份和隐私承诺
  - 本地数据管理、任务记录和危险操作要有清晰层级，不要混成一张长表单

#### 5.2.3 动效与手感要求

- 动效时长控制在 `180ms - 240ms`
- 优先使用 `opacity / translate / scale`，避免布局抖动
- 底部抽屉、卡片展开、Hero 切换采用轻量弹性反馈
- 幻灯片和故事播放允许保留更有氛围的动效，但需提供降噪和低运动适配空间

#### 5.2.4 HTML 页面逐页落地规格

说明：
- 以下页面规格直接从 [mobile-ui-wireframes.html](/Users/maoyuan/code/travel-video-llm/my-spec/docs/mobile-ui-revamp-prd/mobile-ui-wireframes.html) 提炼。
- 后续实现不得只“借用风格”或“部分吸收心智”，而必须逐页对照 HTML 的结构、模块顺序、主次按钮、状态表达和交互收纳方式。
- 若现有代码与以下规格冲突，以本节为准；只有在 PRD 明确注明“本轮不做”的页面，才允许不实现完整视觉重构。

页面 01：冷启动欢迎态
- 页面定位：无数据时的唯一完整启动页，不得再拆成多个零散空态卡片。
- 顶部结构：系统状态栏之下直接进入欢迎 Hero，不再出现传统列表头。
- 首屏模块顺序：
  - `MEMORY START` kicker
  - 一句话主标题：自动整理最近的旅行照片为回忆
  - 一段短说明：默认不上图，只同步 metadata 与端侧结果
  - 三条承诺：默认不上图 / 自动聚合事件 / 本机回看与播放
  - 主按钮 `整理最近 200 张`
  - 次按钮 `手动补导入`
  - 底部辅助说明：首次授权后直接进入主链路，手动选图保留但不再是第一入口
- 实现约束：
  - 主 CTA 必须唯一且最重，不得再出现多个竞争按钮
  - 承诺区必须是简短卡片式表达，不能回退成大段说明文案

页面 02：回忆首页
- 页面定位：唯一主入口，首屏同时承接最近回忆、整理状态和回忆流。
- 顶部结构：
  - 左侧大标题 `回忆`
  - 右侧轻量状态胶囊 `整理中 N`
- 首屏模块顺序：
  - 顶部状态条：后台仍在继续整理，右侧为 `查看`
  - 最近回忆 Hero 卡：封面图、地点、日期、照片数、简短摘要、主动作 `继续回看`、次动作 `播放回忆`
  - 回忆流筛选：全部 / 整理中 / 待更新
  - 按月份组织的回忆流卡片

## 7. 方案约束

- 必须遵守的业务约束：
  - 当前产品主方向仍是“单设备、隐私优先、默认不上图”
  - 自动导入是主入口，手动补导入是次级入口
  - 地图不是主入口，不能再占据第一视图心智
  - 用户中心仍然是设备与数据管理中心，不是账号中心
- 性能约束：
  - 长列表继续使用虚拟化列表，不因设计重构退回低性能实现
  - 大图 Hero、地图页、播放器动效必须控制渲染压力
  - 页面切换应优先保持顺滑，而不是堆积视觉装饰
- 安全 / 隐私约束：
  - 隐私承诺需要清晰表达，但不能在每个页面重复轰炸用户
  - 危险操作必须有清晰二次确认
- 兼容性约束：
  - 不以历史兼容为优先目标
  - 当前不符合产品方向的旧页面心智、旧文案、旧组件风格可直接删除、替换或重构
  - 第一阶段仅要求浅色模式高质量完成

## 8. 风险与待确认

- 风险 1：
  - 如果视觉过度追求“设计感”，可能会损伤当前产品最重要的清晰度和可操作性。
- 风险 2：
  - 若仍保留太多旧组件样式，最终会变成“半套新设计 + 半套旧设计”的拼接结果。
- 风险 3：
  - 若首页同时承载回忆、任务、导入、筛选、空态，信息排布稍有失衡就会再次变乱。
- 还需要确认的问题：
  - 首页主标题最终命名是否使用 `回忆`，还是保留更功能化的 `旅程`
  - 第一阶段是否引入自定义中文标题字体，还是先完全使用系统字体
  - 任务中心首页摘要是否需要在开发阶段进一步压缩一层信息密度

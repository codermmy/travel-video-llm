## 1. 背景

- 当前移动端已经有一轮 UI 重构代码，但经重新复审后，确认**尚未真正对齐 HTML 设计稿**。
- 当前问题不是单个样式偏差，而是：
  - 页面结构虽然“像改过”，但并没有严格按页面 01~15 的模块顺序和层级落地。
  - 首页、地图页的“整理中 / 分析中”状态存在**假状态**，会出现“任务已完成但首页仍显示分析中”的问题。
  - Tab 壳层仍残留旧路由心智，未彻底收口到 `回忆 / 地图 / 我的`。
  - 事件详情页仍残留多余指标和说明文案，没有完成设计稿要求的减法。
  - 之前把 `A1-D4` 勾成完成是不真实的，当前必须按“返工校正”重新推进。

## 2. 本次返工目标

- 不是继续微调现有 UI，而是**把当前实现重新对齐到 HTML 设计稿和原始 PRD**。
- 本次返工的目标不是“更像设计稿”，而是：
  - 页面结构尽量按 HTML 1:1 对齐
  - 视觉节奏、模块顺序、留白、按钮权重、状态呈现尽量按 HTML 1:1 对齐
  - 只有在原生约束或现有真实交互限制下，才允许非 1:1 偏离
- 本次返工不再只盯几个关键页，而是把 **页面 01~15 全部纳入审计范围**。
- 返工完成后，只有在页面与 HTML 逐页对照通过时，才允许重新勾选原始 PRD 中的 `A1-D4`。

## 3. 复审结论摘要

### 3.1 已确认的硬偏差

- 首页与 Tab
  - `events` 仍以 `(tabs)` 子路由存在，说明壳层没有真正收口。
  - 首页 `整理中 N` 不是按真实任务状态算，而是从事件字段硬推导。
  - 首页回忆流卡片仍偏重，信息层级比设计稿更杂。
- 地图
  - 顶部仍偏“标题卡”，不是设计稿里的轻量 floating pill。
  - 地图层存在多余状态浮层或说明层，不够纯净。
  - 聚合列表和单事件抽屉仍偏实现组件，而不是设计稿的页面 03/06 表达。
- 事件详情
  - 首屏仍残留 `照片 / 章节 / 封面` 这类指标层。
  - 状态提示和解释性文案过多，首屏没有做到“只保留必要信息”。
  - 更多操作虽然存在，但详情页整体仍更像资料页，不够像“旅行记忆册”。
- 文档状态
  - 原总 PRD 与 execution PRD 的部分完成勾选过早，需要按真实验收重新处理。

### 3.2 更高一级的返工结论

- 当前问题不只是“几个页面没对齐”，而是**整套 UI 的设计执行标准不够严格**。
- 目前很多页面虽然功能存在，但仍然停留在：
  - “按现有实现顺手排一下”
  - “复用现成组件凑一版”
  - “大致吸收设计稿风格”
- 这不符合本次产品要求。接下来必须改为：
  - **先看 HTML 页面结构**
  - **再按 HTML 重新搭页面**
  - **最后才让真实数据和交互接回去**
- 换句话说：后续返工默认以设计稿为源，而不是以当前代码为源。

## 5. 返工原则

- 先对齐**结构和状态真值**，再对齐视觉细节。
- 不允许继续用“已有类似能力”来替代页面 01~15 的明确设计结构。
- 不允许再根据事件字段猜测“整理中 / 分析中”，必须统一以真实任务状态为准。
- 不允许继续保留“隐藏但还存在”的旧路由心智。
- 在 HTML 明确要求减法的地方，必须真的删掉多余信息层，而不是只是换个样式保留。
- 能 1:1 对齐时，优先 1:1；不能 1:1 时，必须说明偏离原因。
- 不以“当前组件复用方便”为优先级；必要时允许重写页面级 UI。

## 6. 需求拆解

### 6.1 全页面审计要求

- 每个页面都必须建立一份“设计稿 vs 当前实现”的差异清单
- 差异清单至少包含：
  - 模块顺序
  - 页面标题与首屏信息
  - 主次按钮
  - 卡片层级
  - 状态呈现
  - 是否有多余说明文案
  - 是否有设计稿中存在但实现缺失的二级流
- 审计完成前，不允许再把页面视为“已完成”

### 6.2 壳层与首页返工

- Tab 只保留 `回忆 / 地图 / 我的`
- 首页状态胶囊改为读取真实任务状态
- 首页系统 Banner 改为真实任务文案，而不是事件推导文案
- 首页 Hero、状态中心、筛选、回忆流的模块顺序继续按页面 02 收紧
- 回忆流卡片继续做减法，保持媒体优先、低噪音元信息
- 页面 14 的长按快捷操作要继续和 HTML 对齐

### 6.3 地图返工

- 顶部改为页面 03 的 floating pill + 必要工具按钮
- 清除地图层多余悬浮状态层
- 单事件抽屉继续向页面 03 的轻量信息卡收口
- 聚合列表抽屉继续向页面 06 的 cluster sheet 收口
- Marker、筛选、抽屉统一状态语言
- 页面 06 的聚合列表要继续按 HTML 收口，不接受“只是有个列表就算完成”

### 6.4 事件详情返工

- Hero 只保留返回、更多、标题、地点、时间
- 首屏不再保留 `照片 / 章节 / 封面` 指标层
- 主动作严格只保留：
  - `播放回忆`
  - `查看照片`
  - `更多`
- 轻状态条保留，但解释性文案继续压缩
- 故事引子、章节、相册、完整故事保持固定顺序
- 页面 15 的封面选择 / 移动目标选择必须是明确可见的二级流

### 6.5 照片与导入链路返工

- 页面 07、08、09、13 要重新对照 HTML 做视觉审计
- 照片查看器继续压到更接近设计稿的沉浸式信息层
- 照片管理页继续对齐选择态、批量操作和底部动作布局
- 手动补导入选择器继续对齐相册式选择体验
- 权限恢复模板要统一到所有入口，不允许出现另一套权限拒绝样式

### 6.6 我的页与任务中心返工

- 页面 05、10、11 也必须重新审计，不因为“功能大体正确”就跳过
- 我的页要继续往设计稿的信任中心表达收紧
- 任务中心要继续压缩信息噪音，确保不是“实现明细堆砌”
- 本机资料和头像来源页要继续按设计稿层级对齐

### 6.7 状态语言返工

- 首页、地图、任务中心、顶部状态条统一读同一套任务真值
- 无运行中任务时，不允许继续展示“整理中 N”
- 待更新状态只在真实 stale 情况下展示
- “分析中 / 整理中 / 待更新 / 已就绪 / 失败”必须有统一映射

### 6.8 文档返工

- 原总 PRD 的勾选状态只保留真实通过项
- 原 execution PRD 的“关联总任务”完成态必须与真实验收一致
- 本返工 PRD 作为本轮整改基线，直到重新验收通过

- Tab 视觉与路由都只剩 `回忆 / 地图 / 我的`

- 最终样式目标是尽量接近设计稿 1:1，而不是“风格差不多”
- 原 PRD checkbox 与实际完成状态重新一致

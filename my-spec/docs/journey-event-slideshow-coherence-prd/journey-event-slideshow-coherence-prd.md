# 需求文档：旅程页、事件编辑与视频模板系统改造

- 文档状态：Confirmed
- 优先级：P0
- 负责人：
- 创建时间：2026-04-03
- 最后更新：2026-04-03

## 1. 背景

- 当前现状：
  - 产品主链路已经转向“最近照片导入 -> metadata 去重 -> 自动聚合事件 -> 端侧结构化结果 -> 云端故事 -> 端侧幻灯片/导出”。
  - `旅程`页当前仍保留较重的 dashboard 心智，顶部存在“已整理照片 / 最近整理 / 待更新故事”三张指标卡，主信息不是事件内容，而是系统状态。
  - `我的`页当前存在重复信息与重复入口，例如“导入记录数字卡”和“查看导入任务”是分离的两个入口，“产品原则”又重复解释了已经在导入链路中体现的口径。
  - 事件编辑链路已具备部分版本与 stale 标记能力，但还没有把“event/photo 变化会如何影响 story / chapter / slideshow / export”定义成完整的一致性规则。
  - 幻灯片播放器与导出器当前把 `photo / collage / chapter-intro / chapter-summary` 画在不同坐标体系里，导致播放和导出时存在明显的视觉跳动，不符合视频成片要求。
  - 当前成片仍更像“照片播放器页面的导出结果”，而不是“完整视频作品”，包括：
    - 导出比例决策不完整
    - 字幕并非完全属于视频画面
    - 画面外仍在讨论播放器背景，而不是标准规格视频画布
- 为什么要改：
  - 当前页面信息密度和解释性文案过多，抢占了用户对旅程内容本身的注意力。
  - 当前幻灯片的 scene 布局不统一，用户已经明确感知到“机位跳动”“字幕线不一致”“横竖屏适配和卡片位置不统一”的问题。
  - 当前事件与照片的增删改查，只处理了部分 stale 标记，没有把所有受影响的派生结果统一纳入刷新口径，存在故事、章节、幻灯片、导出结果错位或过期的风险。
- 触发这次变更的原因：
  - 用户明确提出以下问题：
    - `我的`页中“产品原则”应删除，隐私和产品口径应只在导入与分析链路中体现。
    - “导入记录 + 数字”本身就应是导入历史入口，不应再额外做一个“查看导入记录/导入任务”的平级入口。
    - `旅程`页 UI 过丑、信息层级不清晰，需要按顶尖移动产品标准重做信息架构与视觉。
    - 事件内照片调整后，横竖屏主导、故事、章节、视频适配都需要严谨重算，不能只局部刷新。
    - 幻灯片中的序幕、卡片、拼图、照片应该共享统一的舞台和字幕线，而不是不同 scene 各用一套位置。
    - 产品最重要的目标是“生成完整规格的视频作品”，不是“在 App 里看一个照片播放器”。

## 2. 目标

- 本次要解决什么问题：
  - 让 `旅程` 页从“系统状态汇总页”回归为“旅行事件浏览页”。
  - 让 `我的` 页从“原则说明页”回归为“本机资料与导入记录管理页”。
  - 明确 event/photo 编辑后，哪些数据是源数据，哪些内容是派生结果，以及这些派生结果何时 stale、何时需要重算。
  - 把幻灯片预览与视频导出统一到一套视觉布局契约，避免 scene 之间位置跳动。
  - 把当前“幻灯片/播放器”体验升级成“视频优先”的产品定义，让系统输出的是一支标准规格的完整视频作品。
- 预期结果是什么：
  - 用户进入 `旅程` 页时，第一眼看到的是事件内容，不是大量说明文字和指标卡。
  - 用户进入 `我的` 页时，能快速看到本机资料、导入历史、清理动作，不再看到重复原则说明。
  - 用户对事件执行增删改查、移动照片、导入新照片、端侧识别补齐后，系统能稳定地把所有相关结果标记 stale 并在适当时机重算。
  - 视频预览与导出都采用统一的标准视频画布，序幕、拼图、单张照片、尾声共用统一比例、统一字幕线、统一镜头语言。
- 成功标准是什么：
  - `旅程` 页顶部不再出现 3 张全局指标卡。
  - `我的` 页不再出现“产品原则”版块，不再存在两个并列的导入历史入口。
  - 任一事件内照片集合变化后，受影响事件的故事、章节、幻灯片状态都会进入明确的待更新状态。
  - 幻灯片预览与导出都采用统一的 scene 舞台、统一字幕线、统一内容安全区。
  - 一条视频只允许一个最终比例，不在 scene 之间切换比例。
  - 非匹配比例的照片直接使用黑色留白，不做播放器式铺底背景。
  - 用户给出的典型场景“2 竖 + 10 横”在事件调整后不会继续沿用过期的竖屏主导布局。

## 3. 非目标

- 这次明确不做什么：
  - 不重做自动聚类算法本身。
  - 不重做端侧识别模型本身。
  - 不在本次引入复杂的事件合并助手、拆分向导、撤销/重做历史。
  - 不在本次改成服务端上传原图的产品方案，继续坚持默认不上图。
  - 不在本次把 `旅程` 页做成复杂的运营后台或分析仪表盘。
- 哪些相关问题先不处理：
  - 不在本次重新定义 AI 故事 prompt 的文风。
  - 不在本次做跨设备同步或账号体系。
  - 不在本次做超过两套的风格主题系统。
  - 不在本次引入 scene 级比例切换、复杂智能裁切、或多种实验性特效模板。

## 4. 用户场景

- 用户是谁：
  - 已导入旅行照片、希望快速回看事件和生成视频的普通用户。
  - 对自动聚类不完全信任，需要手动修正照片归属和事件边界的用户。
  - 对视频观感敏感，希望生成内容具有稳定镜头语言和高级感的用户。
- 典型使用场景：
  - 用户首次导入最近照片，系统生成若干事件，用户在 `旅程` 页浏览事件卡片。
  - 用户发现某张照片被聚错了，进入事件详情把照片移到另一个事件，或从事件中移出若干张新建事件。
  - 用户新增照片到某事件后，希望故事、章节、幻灯片和导出自动跟着当前版本更新。
  - 用户打开幻灯片播放，期望所有 scene 在统一机位和字幕线下连续播放，而不是每种 scene 都像换了一个模板。
- 关键操作路径：
  - 导入最近照片 -> 查看 `旅程` 列表 -> 进入事件详情 -> 调整照片归属 / 编辑事件 -> 系统自动重算 -> 播放幻灯片 / 导出视频。

## 5. 需求拆解

### 5.1 功能需求

- 需求 1：重构 `旅程` 页信息架构
  - `旅程` 页顶部应只保留轻量标题区，不再保留全局 dashboard 式指标卡。
  - 标题区信息应简化为：
    - 页面标题：`旅程`
    - 一行弱信息：例如 `12 个事件 · 186 张照片`
    - 可选的一条轻量状态提示：例如 `3 个事件待更新`
  - `最近整理` 不再作为独立指标卡出现。
  - `待更新故事` 不再作为独立全局指标卡出现，而应下沉为事件级状态。
  - `已整理照片` 如保留，只能作为标题区的一部分弱信息，不再占一个独立卡片。
  - 事件列表是页面主角，进入页面后应尽快看到按月分组的事件卡片。

- 需求 2：重构 `我的` 页信息架构
  - 删除“产品原则”版块。
  - “导入记录 + 数字”本身应成为导入历史入口，不再额外放一个平级“查看导入任务/导入记录”入口。
  - 删除“最近入口 / 最近 200”这类弱价值统计。
  - `我的` 页应主要承载：
    - 本机资料
    - 导入历史
    - 本地清理
    - 必要的设备身份信息
  - 隐私与默认不上图说明只应出现在导入决策、进度反馈、异常提示等真正需要用户理解的位置。

- 需求 2A：为 `我的` 页补齐详细 UI 设计规范
  - 设计目标：
    - 页面气质应接近顶尖移动产品中的“高信任、低打扰、强秩序”个人中心，而不是营销页或后台面板。
    - 视觉上参考微信等成熟产品的优点，但不直接复制具体样式；重点学习其信息层级、留白、分组、列表行节奏和高频操作的稳定感。
  - 设计系统方向：
    - 页面模式：`Minimal Single Column`
    - 风格方向：高信任原生极简，不做夸张拟物、厚重渐变或过多装饰卡片
    - 关键词：`trusted / native / minimal / private / calm / precise`
    - 页面应优先服务“扫描效率”和“操作确定性”，而不是展示概念文案
  - 视觉语言：
    - 采用单列纵向信息流，减少并列统计卡
    - 主背景保持浅色、克制、温和，延续当前产品暖米色系，但降低大面积渐变存在感
    - 重点色只用于 CTA、状态点和少量高亮，不大面积铺满 hero 区
    - 图标统一、线性、低噪音，不使用过度拟态或装饰性图标
    - 推荐 token 方向：
      - 背景：暖白 / 浅米
      - 主文字：深墨黑
      - 次级文字：中灰绿或暖灰
      - 主操作色：延续当前深绿系
      - 微高亮：可少量使用低饱和金色或暖棕，只用于数字或状态，不用于整块铺色
    - 组件表面建议使用 grouped list 风格：
      - 分组圆角：14-18
      - 行高：56-64
      - 外层水平边距：16-20
      - 分组垂直间距：20-28
      - 头像尺寸：64-72
  - 页面结构：
    - 顶部为轻量身份区：
      - 头像
      - 昵称 / 本机名称
      - 一行弱说明，例如“这台设备上的旅行记忆”
    - 第一组为“导入记录”：
      - 该组是页面最重要的功能组
      - 组首主行显示“导入记录”与数字
      - 该主行本身可点击进入导入历史
      - 组内弱信息显示最近导入时间、最近尝试时间
    - 第二组为“本机资料”：
      - 编辑昵称等轻量资料
      - 本机标识、注册时间等弱信息
    - 第三组为“本地清理”：
      - 清理导入记录作为危险动作单独放在页面下部
      - 与普通导航项分组隔离，避免误触
  - 组件表达：
    - 主入口使用原生列表行，而不是大块说明卡
    - 每一行只承载一个主动作或一类信息
    - 行内结构建议为：
      - 左侧图标或头像
      - 中间标题 + 单行副文案
      - 右侧值或箭头
    - “导入记录 + 数字”这一行可使用更强字重与更高触达权重，但仍保持列表行范式，不做夸张 KPI 卡
  - 文案策略：
    - 删除“产品原则”“当前口径”这类说明型标题
    - 删除长段产品介绍
    - 文案统一改为动作与结果导向，例如：
      - `导入记录`
      - `最近导入`
      - `最近尝试`
      - `编辑本机资料`
      - `清理导入记录`
    - 隐私口径仅保留为短句提示，在确有必要处出现，不常驻占主视觉
  - 信息层级：
    - 一级：昵称、导入记录入口、清理动作
    - 二级：数字、时间、资料项
    - 三级：设备标识、解释性说明
    - 不允许一级和三级信息使用同等字号和视觉重量
  - 排版要求：
    - 标题区和分组之间保留明确留白
    - 正文字号采用稳定 scale，不使用零散字号
    - 推荐层级：
      - 页面主标题：28-32
      - 分组标题：17-20
      - 行标题：16-17
      - 副文案：13-14
      - 弱说明：12-13
    - 行高保持充足，保证高端、安静、可扫读
    - 数字与时间使用稳定对齐方式，避免页面视觉抖动
  - 交互要求：
    - 所有可点区域满足最小触控尺寸
    - 点击行时有轻量高亮和按压反馈
    - 异步动作按钮在执行中必须禁用并显示清晰反馈
    - 清理动作必须二次确认
  - 状态设计：
    - Loading：采用轻量 skeleton 或安静 loading，不使用大段加载说明
    - Empty：
      - 若暂无导入历史，也应保持信息结构稳定
      - 不因空状态引入新的营销文案块
    - Error：
      - 错误信息简洁直白
      - 提供明确重试按钮
  - 动效要求：
    - 过渡应克制，时长控制在轻量范围内
    - 以 opacity / translate 为主，不做夸张缩放
    - 重点是“稳定”和“顺滑”，不是“炫”
    - 建议将身份区、分组卡片、列表行动画统一在同一套节奏内，避免每块各自表演
  - 反模式：
    - 不要再出现 3 张并列统计卡
    - 不要在 `我的` 页堆长段原则说明
    - 不要把“导入记录入口”和“导入记录数字”拆成两个互相竞争的入口
    - 不要让危险操作与普通导航项混排
    - 不要把页面做成渐变海报或账号营销中心

- 需求 3：明确 event/photo 是源数据，story/chapter/slideshow/export 是派生结果
  - 以下内容应被定义为源数据：
    - 照片与事件的归属关系
    - 照片顺序
    - 照片 metadata（时间、GPS、宽高、文件大小等）
    - 照片端侧识别结果
    - 事件上参与叙事输入的字段，如地点、结构化标签
    - 用户手工维护字段，如事件标题
  - 以下内容应被定义为派生结果：
    - 事件摘要信息：时间范围、封面、中心点、照片数量
    - 事件 composition profile：横竖屏主导、舞台布局适配方向
    - 章节 `chapters`
    - 照片组 `photoGroups`
    - 故事文本 `story / full_story`
    - 幻灯片 scene timeline
    - 导出视频缓存与视频成品

- 需求 4：定义 event/photo 变化后的统一重算规则
  - 以下行为必须视为会影响派生结果：
    - 事件新增照片
    - 事件移除照片
    - 照片从事件 A 移到事件 B
    - 选中若干照片新建事件
    - 删除事件并使照片进入无事件状态或归入其他事件
    - 修改会进入故事 prompt 或章节摘要的事件字段
    - 影响横竖屏判断、章节切片、字幕内容的照片宽高或端侧识别结果变化
  - 以下行为默认不触发故事与幻灯片重算：
    - 仅修改事件标题
    - 仅更换事件封面
    - 仅修改不参与 prompt 的纯展示字段
  - 当照片从事件 A 移到事件 B 时，A 和 B 都必须视为受影响事件，两边都要重算。
  - 当端侧识别结果晚于事件编辑到达时，如果这些识别结果仍属于当前版本输入的一部分，系统必须让受影响事件重新进入待更新状态。

- 需求 5：定义统一的版本口径
  - `event_version` 应重新定义为“事件派生输入版本”，而不仅仅是“结构变化版本”。
  - 任何会影响故事、章节、幻灯片、导出的源数据变化，都应让 `event_version + 1`。
  - `story_generated_from_version` 表示当前故事基于哪个事件输入版本生成。
  - `slideshow_generated_from_version` 表示当前幻灯片/导出布局基于哪个事件输入版本生成。
  - 若异步任务返回的是旧版本结果，则不得覆盖当前最新结果。

- 需求 6：补齐 composition profile 与布局适配逻辑
  - 每个事件都应基于当前照片集合重新计算当前的 composition profile。
  - composition profile 的计算至少要依赖可靠的照片宽高数据。
  - 事件调整后不得继续沿用旧事件版本的横竖屏主导判断。
  - 典型场景“2 张竖图 + 10 张横图”在移动照片后，应以最新集合重新判断，不得继续沿用旧的竖屏主导布局。

- 需求 7：确立“视频优先”产品定义
  - 系统的主产物是“一支标准规格的完整视频作品”，而不是“照片播放器页面”。
  - App 内播放页的角色应降级为“成片预览器”，其使命只是预览最终视频，而不是单独存在一套播放器设计。
  - 预览必须尽量等价于导出；用户在 App 里看到的构图、字幕和黑色留白策略，不应与导出的 mp4 明显不一致。

- 需求 8：统一视频比例决策
  - 一条视频只允许一个最终输出比例，不允许 scene 级切换比例。
  - 默认导出比例策略：
    - 横图主导事件：`16:9`
    - 竖图主导事件：`9:16`
    - 混合事件：`Auto`，按主导方向判定
  - 导出与预览都应支持：
    - `Auto`
    - `16:9`
    - `9:16`
  - 用户临时选择导出比例属于渲染偏好，不属于事件内容版本，不应进入 `event_version`。

- 需求 9：统一视频模板系统与布局契约
  - `photo / collage / chapter-intro / chapter-summary` 必须共享同一套内容舞台坐标，而不是每类 scene 都单独定义一套页面布局。
  - 所有 scene 必须共享：
    - 统一视频画布 `canvas`
    - 统一标题安全区 `title safe area`
    - 统一媒体舞台 `scene stage`
    - 统一字幕安全区 `subtitle safe area`
  - 横图、竖图、拼图的差异，只允许发生在“舞台内部如何适配素材”，不允许改变整个 scene 的机位。
  - 幻灯片预览与视频导出必须消费同一份 layout contract，不允许播放器一套、导出器一套。
  - 最终落地目标是“标准视频模板系统”，不是“美化后的播放器页面”。

- 需求 10：将“卡片”改造成视频镜头语言
  - 不再把序幕、拼图、章节、照片表达为 App UI 卡片。
  - 视频内只允许三类视觉单元：
    - `Title Plate`：用于序幕、章节、尾声
    - `Montage Frame`：用于双图、三图、多图拼贴
    - `Photo Frame`：用于单张照片展示
  - `Title Plate` 应以 Typography-first 为核心，不使用厚重边框、阴影、毛玻璃 UI 卡片。
  - `Montage Frame` 应更接近 contact sheet / triptych，而不是 UI grid 卡片。
  - `Photo Frame` 应让照片成为主角，不再由明显 UI 容器包裹。

- 需求 11：统一字幕系统
  - 字幕永远属于视频画面内部，不允许在画面下方外挂一个独立容器。
  - 所有 scene 共用同一条字幕基线。
  - 字幕上限 2 行。
  - 16:9 与 9:16 只允许调整字号和基线，不允许换一套字幕系统。
  - 字幕可读性应通过轻阴影、轻描边或极淡字幕带解决，禁止使用粗重字幕条。

- 需求 12：废弃默认模糊铺底，改用电影化背景策略
  - 默认禁止“全屏模糊照片铺底”。
  - 非匹配比例的留白区域默认直接采用纯黑背景。
  - 16:9 视频中的竖图，应表现为左右黑色留白。
  - 9:16 视频中的横图，应表现为上下黑色留白。
  - 这些黑色留白属于标准视频画布的一部分，不应再被视为需要“补背景”的问题。

- 需求 13：减少页面旁白，提升顶级产品化表达
  - `旅程` 页、`我的` 页、事件详情页都应减少解释系统在做什么的大段旁白。
  - 只保留真正有决策价值的文案：
    - 当前可执行动作
    - 当前状态
    - 当前异常
  - 产品口径说明应从“长期常驻说明块”改为“在需要时才出现的上下文提示”。

### 5.2 交互 / 界面要求

- 页面或入口：
  - `旅程` 页
  - `我的` 页
  - 事件详情页
  - 幻灯片播放器
  - 视频导出链路
- 用户操作：
  - 用户从 `旅程` 页直接浏览事件列表，不先阅读指标卡和说明文案。
  - 用户从事件卡片或事件详情进入编辑、管理照片。
  - 用户调整照片后，页面应立即反馈“内容待更新 / 正在整理 / 生成中 / 已完成”。
  - 用户播放视频预览时，不应感知 scene 之间的视觉机位跳变。
  - 用户应明确感知自己在看“一支视频的预览”，而不是照片浏览器。
  - 用户在导出时可以选择 `Auto / 16:9 / 9:16`，但系统必须先给出稳定默认值。
  - 用户进入 `我的` 页时，应先看到身份和导入历史主入口，而不是原则说明。
- 状态反馈：
  - 事件级状态优先，页面级状态弱化。
  - 事件状态至少要区分：
    - `等待端侧分析`
    - `待更新`
    - `生成中`
    - `已完成`
    - `生成失败`
  - 若列表中存在多个 stale 事件，可在页面顶部显示一条很薄的聚合提示，但不做 3 张指标卡。
  - 视频 stale 时，应明确告诉用户当前内容是否仍可预览、是否允许导出。
- 异常提示：
  - 目标事件不存在、照片不存在、照片不属于当前用户时，要给出明确错误。
  - 若缺少可靠照片尺寸而无法稳定判断横竖屏，应有明确降级策略，而不是静默采用错误布局。
  - 若当前事件仍在等待端侧分析，应提示结果会在完成后自动更新。
  - 若当前视频仍是旧版本，应允许预览但默认禁止导出，并明确提示“内容待更新，可先预览旧版本”。

### 5.3 数据 / 状态要求

- 需要新增或调整的数据：
  - 事件版本号 `event_version`
  - 当前故事基于的版本 `story_generated_from_version`
  - 当前幻灯片基于的版本 `slideshow_generated_from_version`
  - 故事新鲜度 `story_freshness`
  - 幻灯片新鲜度 `slideshow_freshness`
  - 事件是否存在待更新派生结果 `has_pending_structure_changes`
  - 照片可靠宽高字段（服务端持久化）
  - 事件当前 composition profile（可计算可缓存）
  - 视频比例模式 `video_aspect_mode`：
    - `auto`
    - `landscape`
    - `portrait`
  - 解析后的最终比例 `resolved_video_aspect_ratio`
  - 统一视频模板 contract：
    - `canvas`
    - `title_safe_area`
    - `stage_rect`
    - `subtitle_safe_area`
  - 可选：待更新原因枚举，例如 `photo_membership_changed / vision_updated / layout_profile_changed`
- 状态流转：
  - 源数据变化：
    - `event_version + 1`
    - `story_freshness = stale`
    - `slideshow_freshness = stale`
    - `has_pending_structure_changes = true`
  - 若当前事件存在 `pending / processing` 照片：
    - 事件状态为 `waiting_for_vision`
  - 若当前事件已满足故事生成条件：
    - 事件状态为 `ai_pending -> ai_processing -> generated / ai_failed`
  - 若故事已基于当前版本生成成功：
    - `story_generated_from_version = event_version`
    - `story_freshness = fresh`
  - 若幻灯片布局已基于当前版本生成成功：
    - `slideshow_generated_from_version = event_version`
    - `slideshow_freshness = fresh`
- 持久化要求：
  - 服务端需要持久化照片宽高，不能只在导入 schema 中存在而不落库。
  - 服务端需要持久化事件版本、新鲜度、故事版本、幻灯片版本。
  - 客户端播放器与导出器需要消费同一套布局 contract，不允许分别维护两套硬编码坐标。
  - 用户临时导出选择的比例不应写入 `event_version`；若后续需要记忆事件级偏好，应作为独立偏好字段处理。
  - 默认不上图原则继续保持，云端不持久化原图。

### 5.4 变更影响矩阵

- 变更类型：向事件新增照片
  - 受影响事件：目标事件
  - 必须重算：事件摘要、composition profile、chapters、photoGroups、story、slideshow、export
  - 备注：若新增照片来自其他事件，则原事件也属于受影响事件

- 变更类型：从事件中移除照片
  - 受影响事件：原事件
  - 必须重算：事件摘要、composition profile、chapters、photoGroups、story、slideshow、export
  - 备注：若原事件照片被清空，应按既定规则删除事件或进入无内容态

- 变更类型：照片从事件 A 移到事件 B
  - 受影响事件：事件 A、事件 B
  - 必须重算：A/B 两边的事件摘要、composition profile、chapters、photoGroups、story、slideshow、export
  - 备注：这是最容易出错的高优先级场景，必须两边同时 stale

- 变更类型：选中若干照片新建事件
  - 受影响事件：原事件、新事件
  - 必须重算：原事件与新事件的事件摘要、composition profile、chapters、photoGroups、story、slideshow、export

- 变更类型：照片端侧识别结果从 `pending/processing` 进入终态，或识别内容被修正
  - 受影响事件：该照片所属事件
  - 必须重算：事件状态、chapters、photoGroups、story、slideshow 文案层
  - 备注：若识别结果还会影响布局或封面选择，也必须同步使 slideshow stale

- 变更类型：照片宽高或方向信息补齐/修正
  - 受影响事件：该照片所属事件
  - 必须重算：composition profile、slideshow、export
  - 默认不重算：story 文本
  - 备注：这是解决“事件已改，但布局仍沿用旧横竖屏判断”的关键场景

- 变更类型：修改事件地点、结构化标签等参与 prompt 的字段
  - 受影响事件：当前事件
  - 必须重算：story、chapters、photoGroups
  - 可选重算：slideshow 文案层

- 变更类型：修改事件标题
  - 受影响事件：当前事件
  - 直接更新：列表标题、详情标题、播放器标题、导出标题
  - 默认不重算：story、chapters、slideshow 布局
  - 备注：标题应视为用户覆盖字段，不再被后续 AI 自动改写

- 变更类型：修改事件封面
  - 受影响事件：当前事件
  - 直接更新：列表封面、详情封面
  - 默认不重算：story、chapters、slideshow、export

- 变更类型：修改音乐
  - 受影响事件：当前事件
  - 直接更新：播放器与导出音轨选择
  - 默认不重算：story、chapters、composition profile

### 5.5 视频模板视觉规范

- 视觉基线：
  - 先定义“标准规格视频画布”，再定义其上的排版风格
  - 视频画布默认是纯黑背景
  - 非匹配比例的素材直接通过黑色留白适配，不再引入播放器式背景方案
- 排版风格：
  - 在标准视频画布之上，文字与镜头语言采用 `Editorial Cinema` 方向
  - 关键词：
    - `editorial`
    - `refined`
    - `cinematic`
    - `high-trust`
- 画布原则：
  - 一条视频只有一个画布比例
  - 一条视频只有一套镜头语言
  - 序幕、章节、拼图、照片、尾声都必须服从同一套视频模板
- 背景策略：
  - 默认背景使用纯黑
  - 禁止使用默认全屏强模糊原图铺底
  - 禁止把黑色留白再次包装成播放器背景或毛玻璃 UI
- 排版原则：
  - 采用 Typography-first，而不是 UI card-first
  - 标题、章节、地点、日期等信息应被视为“画面内排版”，不是“卡片内容”
  - 同一条视频中标题字体、字幕字体、字号层级必须稳定
- 字体方向：
  - 标题推荐方向：
    - `Libre Bodoni`
    - `Cormorant`
  - 正文 / 字幕推荐方向：
    - `Inter`
    - `Public Sans`
- 色彩方向：
  - 主背景：`#000000`
  - 次背景：`#1C1917`
  - 主文字：`#F8F8F2`
  - 微强调色：`#CA8A04`
  - 原则：高对比、低饱和、少量强调
- Title Plate 规范：
  - 用于序幕、章节、尾声
  - 主标题为主要视觉元素
  - 允许小行 eyebrow 和次级元信息
  - 不允许厚重圆角卡片、明显描边面板、厚阴影
- Montage Frame 规范：
  - 用于双图、三图或多图拼贴
  - 更接近 contact sheet / triptych
  - 只允许细 gutter，不允许明显 UI 卡片边框
  - 双图默认左右对开；三图默认左大右上下
- Photo Frame 规范：
  - 用于单张照片
  - 照片必须是主角
  - 默认 `contain`
  - 不匹配比例留白直接交给黑色画布
- 16:9 布局原则：
  - 更克制、更像电影
  - 标题区在画面中上部
  - 字幕更低、更细、更轻
  - 左右黑色留白可作为明确风格接受
- 9:16 布局原则：
  - 仍保持电影化，而不是短视频模板感
  - 标题位置略上移
  - 字幕位于画面底部约 `1/4` 区域内的固定安全线
  - 双图、三图拼贴强调纵向节奏
- 明确反模式：
  - `Liquid Glass`
  - `Bento Grid`
  - `Parallax Storytelling`
  - 全屏模糊铺底
  - 为了“填满屏幕”而牺牲标准视频画布规则
  - 厚重 UI 卡片
  - scene 之间随机切换机位、字体、边框、阴影

## 7. 方案约束

- 必须遵守的业务约束：
  - 当前产品方向不以历史兼容为优先。
  - 默认不上图原则不变。
  - 自动聚合事件是主链路，手动补导入只是补充能力。
  - 事件编辑是主链路的一部分，不是后台边缘功能。
  - 视频系统必须遵守 `Editorial Cinema` 主风格，不得回退到 UI 卡片式播放器心智。
- 性能约束：
  - 事件编辑后不应全局重算所有事件，只重算受影响事件。
  - 幻灯片布局计算应尽量复用已知宽高和已知状态，避免无必要全量阻塞。
  - 导出与播放器不应各自维护一套复杂布局计算导致双倍维护成本。
- 安全 / 隐私约束：
  - 默认导入链路仍只上传 metadata 与端侧结构化结果。
  - 不能因为这次重构引入原图上云的新隐私风险。
- 兼容性约束：
  - 需要兼容当前已有的事件数据和故事数据。
  - 若新增照片宽高持久化字段，需要考虑旧数据回填或运行时补齐策略。
  - `我的` 页重构应尽量复用现有导航与数据读取逻辑，重点替换布局与信息层级，不引入不必要的新业务状态。

## 8. 风险与待确认

- 已确认口径：
  - 已确认 1：产品输出的是标准规格视频，不是播放器
  - 已确认 2：默认彻底放弃全屏模糊铺底
  - 已确认 3：非匹配比例素材直接使用黑色留白，不做额外背景包装
  - 已确认 4：不再使用“卡片”概念，统一改为 `Title Plate / Montage Frame / Photo Frame`
  - 已确认 5：一条视频只允许一个比例，不做 scene 级切换
  - 已确认 6：默认比例策略为：
    - 横图主导事件：`16:9`
    - 竖图主导事件：`9:16`
    - 混合事件：`Auto`

- 风险 1：
  - 如果只改 UI 不改派生结果版本规则，会继续出现“页面看起来新了，但故事/幻灯片仍吃旧数据”的错乱问题。
- 风险 2：
  - 如果播放器和导出器继续各自维护不同的布局坐标，预览和导出结果会长期不一致，后续每次改布局都要双改。
- 风险 3：
  - 如果照片宽高仍不持久化，只依赖本地运行时数据，则事件横竖屏主导判断在跨会话、导出、后端返回时都不稳定。
- 风险 4：
  - 如果端侧识别结果变化不进入事件版本体系，故事和章节可能在“照片集合没变但内容已变”的情况下继续沿用旧结果。
- 还需要确认的问题：
  - 待确认 1：当事件的 `story_freshness/slideshow_freshness = stale` 时，是否允许继续播放旧视频预览？
    - 推荐口径：允许预览旧内容，但明确标记“内容待更新”；导出按钮默认禁用，避免导出过期成片。
  - 待确认 2：照片端侧识别结果更新，是否应视为需要 `event_version + 1` 的源数据变化？
    - 推荐口径：是。只要识别结果会影响章节、字幕、故事、布局，就必须 bump 版本。
  - 待确认 3：`旅程` 页在已有内容后，是否仍保留“导入最近 200 张 / 手动补导入”的常驻入口？
    - 推荐口径：不常驻占主视觉，只在空状态显示；有内容后移入次级入口。
  - 待确认 4：旧事件若缺少服务端宽高数据，第一版如何补齐？
    - 推荐口径：优先从本地 registry 读取并回写；无法补齐时按保守策略降级，但必须提示这是降级结果。
  - 待确认 5：`has_pending_structure_changes` 是否沿用当前字段名，还是改为更准确的 `has_pending_derivative_refresh`？
    - 推荐口径：如果涉及后端迁移成本可接受，建议改名，否则保留字段名但更新语义。
  - 待确认 6：composition profile 的主导判断是否继续按数量占比，还是升级为更严格的加权规则？
    - 推荐口径：第一版先按可靠宽高的方向数量占比做判断；若方向接近，再结合最近稳定布局或更细规则做决策。

## 11. 发布影响

- 是否需要迁移数据：
  - 需要，若服务端新增照片宽高字段或调整事件状态字段语义。
- 是否影响现有用户：
  - 会明显改变 `旅程` 页、`我的` 页、事件详情页状态表达，以及幻灯片/导出观感。
- 是否需要重新 build / 安装：
  - 需要移动端重新 build；若服务端模型调整则需要后端迁移与部署。
- 回滚方式：
  - 前端页面布局可按模块回滚；后端版本字段与照片尺寸字段若已迁移，需要保留兼容读取逻辑后回滚页面逻辑。

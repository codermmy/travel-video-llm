# 单设备隐私优先产品定位与需求文档（PRD Final）

> 状态：已定稿（待进入技术方案设计与实现拆解）
>
> 最后更新：2026-03-31
>
> 目的：明确当前阶段关于“单设备、默认不上图、端侧生成、显式增强入口”的正式产品需求，作为后续架构设计与实现的输入。

---

## 1. 背景与结论

当前代码虽然已经把查重改成基于 Metadata，但整体链路仍然不是“只传元数据”：

- 前端仍会上传缩略图文件到后端
- 后端仍会保存 `thumbnail_url / object_key / storage_provider`
- 事件故事、章节故事、微故事、事件封面仍依赖可访问的缩略图 URL

因此，如果产品目标是“后端默认不接触用户图片”，那需要重构的不是查重本身，而是：

1. 图片上传策略
2. 视觉理解输入源
3. 故事生成输入契约
4. 幻灯片播放与导出方式

本草案当前已经对齐的产品方向是：

- **产品定位**：单设备、隐私优先、本地生成旅行故事工具
- **默认策略**：图片不上传云端，包含缩略图也默认不上云
- **增强策略**：用户按事件主动上传少量压缩后的代表图或缩略图，触发一次更强故事重生成
- **多设备同步**：本阶段明确不做
- **幻灯片/视频**：默认在端侧基于本地图片播放与导出
- **账号体系**：当前阶段不做面向用户的登录/注册产品能力
- **用户中心**：保留轻量“我的/设置”入口，但定位为本机设置与数据管理，而不是账号中心

## 2. 产品定位

### 2.1 一句话定位

一款以**单设备使用**为前提、以**隐私优先**为默认策略、把本地旅行照片整理成事件、故事与幻灯片的工具。

### 2.2 产品主张

- 默认所有照片留在本机
- 用户不需要先信任云端，才能获得事件整理和故事能力
- 云端增强是一个显式入口，而不是默认模式切换
- 产品首先解决“整理与回看”，再考虑“更强的 AI 叙事”

### 2.3 目标用户

- 有大量旅行照片，但缺乏整理动力的普通用户
- 对照片隐私敏感，不希望默认上传到云端的用户
- 希望快速生成旅行回顾、幻灯片、故事摘要的用户

### 2.4 不再追求的产品方向

以下方向在当前阶段明确降级或放弃：

- 跨设备自动同步照片
- 默认云端存图
- 依赖服务端持有完整图片资源的体验
- 以“云相册”作为核心定位
- 以注册登录体系作为产品主能力入口

## 3. 产品原则

### 3.1 原则一：默认隐私优先

- 用户图片默认不上传
- 用户应能在不上传图片的情况下完成基本事件整理与故事浏览
- 任何图片上传都应是显式授权，而不是静默默认行为

### 3.2 原则二：单设备体验闭环

- 事件生成、图片浏览、幻灯片播放、视频导出应在单设备上形成完整闭环
- 换设备不作为第一阶段核心使用场景

### 3.3 原则三：增强能力按需开启

- 当用户想获得更强故事效果时，再对某个事件主动发起“上传代表图并重新生成”
- 增强应尽量基于少量代表图，而不是全量上传
- 增强入口需要明确说明上传内容、用途和保留策略

### 3.4 原则四：端侧负责图片，服务端负责编排

- 端侧优先负责图片读取、图片筛选、基础识别、幻灯片素材准备
- 服务端优先负责文本编排、故事生成、可控风格输出

---

## 4. 目标产品形态

### 4.1 默认形态：隐私优先本地闭环

默认形态下：

- 照片保留在本地
- 服务端接收 Metadata 与端侧结构化识别结果
- 端侧负责粗粒度图片理解
- 服务端基于 Metadata + 端侧结构化描述生成故事
- 幻灯片基于本地图片播放与导出

用户得到的能力：

- 自动查重
- 自动聚类成事件
- 自动生成事件标题、摘要故事、章节文案
- 自动生成本地幻灯片和本地导出视频
- 事件封面可由本地首图、抽样图或代表图生成

用户失去的能力：

- 跨设备看同一批照片
- 服务端代渲染视频
- 不上传图片情况下的云端强视觉分析

### 4.2 增强入口：用户主动授权上传并重新生成

增强入口下：

- 用户对单个事件主动点击“上传代表图，生成更好的故事”
- 上传少量压缩后的代表图或缩略图，而非整组图片
- 服务端对代表图做更强视觉理解
- 回写更丰富的故事内容、封面建议、章节细节

增强入口的作用：

- 提升故事画面感
- 提升 caption / micro-story 质量
- 提升封面选择质量

### 4.3 账号与设备策略

- 当前阶段产品按单设备使用设计，不承诺多设备同步
- 当前阶段不提供面向用户的注册、登录、找回密码等流程
- 若服务端仍需最小身份标识，可保留设备级身份或匿名 token，但不作为产品功能主入口
- 保留轻量用户中心，用于承载：
  - 本机隐私说明
  - 设备标识展示
  - 本地数据管理
  - 增强素材清理入口
  - 导出与缓存相关设置

## 5. 核心需求文档（PRD Draft）

### 5.1 产品目标

帮助用户在**不默认上传图片**的前提下，把本地旅行照片快速整理成：

1. 旅行事件
2. 事件故事
3. 幻灯片/视频回顾

### 5.2 成功指标

- 用户首次导入后成功生成事件的比例
- 用户首次导入后完成一次幻灯片播放的比例
- 用户首次导入后完成一次本地视频导出的比例
- 用户在默认隐私策略下完成故事生成的比例
- 用户对“是否上传图片”策略的理解度和信任度
- 用户主动使用“上传代表图并重新生成”入口的事件占比

### 5.3 MVP 范围

#### P0：单设备隐私版闭环

- 支持单设备导入照片
- 支持基于 Metadata 的查重
- 支持基于时间/GPS 的事件聚类
- 支持端侧粗粒度图片描述提取
- 支持基于 Metadata + 端侧描述生成基础故事
- 支持本地幻灯片播放
- 支持端侧视频导出
- 支持用户手动选择事件封面
- 不提供用户级登录注册流程
- 保留轻量用户中心/设置页，但不承载账号能力

#### P1：事件增强入口

- 支持对单个事件主动上传少量代表图
- 支持增强后重写故事、章节、caption
- 支持系统推荐封面图

#### 明确不做

- 跨设备自动同步照片
- 默认上传缩略图或原图
- 服务端全量图片存储
- 服务端代生成视频作为 MVP 必备功能
- 多设备同步账号体系

## 6. 核心功能需求

### 6.1 照片导入

**用户故事**

作为用户，我希望导入本机相册中的旅行照片，并在尽量不上传图片的前提下开始整理。

**要求**

- 支持相册授权
- 支持手动选择照片和自动导入最近照片
- 支持基于 Metadata 的查重
- 默认不上传图片到服务端

**验收标准**

- 导入完成后，用户能看到导入数量、去重数量、失败数量
- 未主动触发增强入口时，不发生默认图片上传，包括缩略图

### 6.2 事件生成

**用户故事**

作为用户，我希望系统把同一次旅行/同一段行程的照片自动聚合成事件。

**要求**

- 基于时间/GPS 聚类生成事件
- 允许无 GPS 照片参与事件
- 事件需有标题、时间范围、地点信息

**验收标准**

- 用户导入照片后可看到事件列表
- 每个事件至少有标题兜底、时间范围、照片数量

### 6.3 默认故事生成

**用户故事**

作为用户，我希望即使不上传图片，也能得到基本可读的旅行故事。

**要求**

- 端侧提取粗粒度图片描述
- 服务端消费结构化描述生成故事
- 故事应能输出：
  - 事件标题
  - 事件摘要
  - 章节文案
  - 幻灯片字幕

**验收标准**

- 在未上传图片时，事件仍可生成故事
- 故事质量允许弱于增强版，但应具备可读性和时间地点连贯性

### 6.4 事件封面

**用户故事**

作为用户，我希望每个事件有封面，且我可以自己调整。

**要求**

- 默认可从事件首张、抽样图或代表图中选择封面
- 支持用户手动指定封面
- 在无云端图片场景下，封面优先服务于本机展示

**验收标准**

- 每个事件默认有封面兜底
- 用户可主动更换封面

### 6.5 幻灯片播放与视频导出

**用户故事**

作为用户，我希望在本机直接播放和导出旅行幻灯片，而不依赖云端有图片。

**要求**

- 幻灯片播放基于本地图片
- 字幕使用事件故事、章节文案、微故事
- 视频导出应在端侧完成

**验收标准**

- 单设备上可完成幻灯片播放
- 单设备上可完成本地视频导出
- 未上传图片时仍能播放和导出本地幻灯片

### 6.6 AI 增强入口

**用户故事**

作为用户，我希望在某些重要事件上主动换取更好的故事效果。

**要求**

- 按事件触发，而不是切换全局模式
- 上传 3-5 张压缩后的代表图
- 明确告知“上传了什么、用途是什么”
- 增强完成后回写更丰富的故事结果
- 上传素材短期保留，用于失败重试

**验收标准**

- 用户可明确感知这是一次显式上传并重新生成
- 用户可决定是否上传图片

### 6.7 用户中心 / 设置页

**用户故事**

作为用户，我希望有一个统一入口查看隐私策略、管理本地数据和清理增强素材，但不希望被迫注册登录。

**要求**

- 保留轻量“我的/设置”入口
- 不展示注册、登录、找回密码等账号流程
- 支持查看当前设备身份说明
- 支持查看隐私策略与图片上传说明
- 支持清理缓存、清理增强素材、管理本地导出内容

**验收标准**

- 用户无需登录即可进入该页面
- 用户可以明确理解产品默认不上图
- 用户可以手动清理增强素材和本地缓存

## 7. 数据策略

### 7.1 默认形态下服务端允许持有的数据

- 设备 ID / 匿名用户 ID
- 照片 Metadata
  - `shoot_time`
  - `gps_lat / gps_lon`
  - `file_size`
  - 可选的文件哈希或设备内引用标识
- 端侧结构化视觉描述
  - `scene_category`
  - `object_tags`
  - `activity_hint`
  - `ocr_text`
  - `people_present`
  - `people_count_bucket`
  - `emotion_hint`
  - `landmark_hint`
  - `image_quality_flags`
  - `cover_score`
- 事件和故事结果

### 7.2 默认形态下服务端不应默认持有的数据

- 原始照片
- 缩略图
- 可还原用户视觉内容的图片 URL

### 7.3 增强入口下可临时持有的数据

- 用户主动授权上传的代表图
- 对应的视觉分析结果
- 与增强任务相关的中间结果

### 7.4 数据保留原则

- 默认形态下不保留图片资源
- 增强入口上传的图片与中间结果短期保留
- 默认保留 7 天，用于失败重试与重新生成
- 到期后自动清除
- 用户中心中提供手动清除入口

## 8. 端侧粗识别能力范围建议

第一版不追求端侧大模型写故事，只追求“足够支撑故事编排的粗描述”。

### 8.1 P0 输出字段

- `scene_category`
- `object_tags`
- `activity_hint`
- `people_present`
- `people_count_bucket`
- `emotion_hint`
- `ocr_text`
- `landmark_hint`
- `image_quality_flags`
- `cover_score`

字段说明：

- `scene_category`：如海滩、城市、山地、餐厅、博物馆、夜景
- `object_tags`：图片主体内容标签
- `activity_hint`：如步行、用餐、观景、乘车、合影
- `emotion_hint`：如轻松、热闹、浪漫、安静，仅做粗粒度提示
- `cover_score`：用于代表图与封面候选排序

### 8.2 P1 可扩展字段

- `weather_hint`
- `embedding`
- `fine_grained_place_hint`

### 8.3 不建议在 P0 做的事情

- 端侧直接写长故事
- 端侧直接生成章节文案
- 端侧追求高质量开放式 caption

## 9. 端侧识别技术方向

### 9.1 共同原则

- 端侧只输出结构化字段，不在手机端直接写长故事
- 识别任务以导入后异步批处理为主，避免阻塞主流程
- 优先使用系统级或成熟移动端能力，降低包体、功耗和维护成本
- 统一服务端输入 schema，避免 iOS 和 Android 输出格式分裂

### 9.2 Android MVP 主路线

当前决策：

- **P0 主路线：ML Kit on-device**
  - OCR
  - image labeling
  - 人脸/人物存在检测
  - 质量 flag 计算
- **P1/P2：MediaPipe 或 TFLite**
  - 用于旅行场景定制标签、封面评分、活动识别
- **暂不作为主路径：AICore / Gemini Nano**
  - 机型覆盖和稳定性不足，不适合作为核心链路

### 9.3 iOS 对齐路线

当前倾向：

- **P0 主路线：Apple Vision**
  - OCR
  - 人脸/人体存在检测
  - 粗粒度分类
  - feature print / embedding 用于代表图抽样与封面推荐
- **P1 增强：少量 Core ML 轻量分类模型**
  - 补足旅行场景标签质量
  - 仍然只输出结构化字段
- **暂不考虑：MLX**
  - 不适合当前“粗识别、轻量、稳定”的目标

### 9.4 当前工程判断

- React Native + Expo 若要做稳定端侧识别，基本需要 `Expo Dev Client + 原生模块`
- 不应假设可以长期停留在纯 Expo Go 能力范围内
- 第一阶段先按 Android 路线完成统一输出 schema，再决定是否做自定义模型

## 10. 当前方案的主要优点

- 隐私叙事清晰
- 产品边界明确
- 架构复杂度显著下降
- 幻灯片逻辑更符合实际用户心智
- 允许逐步增强，不必一次性完成最重方案

---

## 11. 当前方案的主要风险

- 放弃多设备同步后，换机迁移体验会变弱
- 默认故事质量会低于“云端看图版”
- 端侧粗识别能力如果过弱，故事会显得空泛
- 本地幻灯片和导出能力会拉高端侧工程复杂度
- Expo 侧为接入原生识别能力，开发流程会从 Expo Go 转向 Dev Client

---

## 12. 已确认的产品结论

- 当前阶段正式放弃多设备同步，产品按单设备设计
- 默认不上传任何图片到云端，包括缩略图
- 幻灯片播放与视频导出都在端侧完成
- 事件封面支持本地首图、抽样图、代表图兜底，且支持用户手动选择
- 不再做面向用户的登录注册产品流程
- 保留轻量用户中心，但仅用于设置、隐私说明和数据管理
- “增强”不是独立模式，而是一个显式上传代表图并重新生成的入口
- 增强入口一次上传 3-5 张压缩代表图
- 增强素材默认短期保留 7 天，到期自动清除
- 默认故事依赖 Metadata + 端侧结构化识别结果，不依赖服务端直接看图

---

## 13. 本版定稿的关键产品决策

- 端侧识别 P0 统一输出字段固定为：
  - `scene_category`
  - `object_tags`
  - `activity_hint`
  - `people_present`
  - `people_count_bucket`
  - `emotion_hint`
  - `ocr_text`
  - `landmark_hint`
  - `image_quality_flags`
  - `cover_score`
- 增强入口一次上传 3-5 张压缩代表图，支持用户手动勾选或系统推荐
- 增强素材与中间结果默认保留 7 天，随后自动清除
- 若故事生成失败，7 天保留窗口内允许直接重试，避免重复上传
- 用户中心保留，但不承载账号体系；建议产品文案使用“我的”或“设置”，避免误导为云账号系统

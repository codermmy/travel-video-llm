# Workstream 04：默认故事生成链路重构

> 角色：后端主需求包
>
> 优先级：P0
>
> 是否允许并行开发：是，但依赖契约与端侧识别字段

---

## 1. 需求背景

当前默认故事生成链路仍是：

`服务端拿缩略图 URL -> 视觉模型识图 -> 生成故事`

这条链路与新产品方向冲突，因为默认版本不应上传图片，也不应让服务端看到图片内容。

---

## 2. 目标

把默认故事链路改造成：

`Metadata + 地理信息 + 端侧结构化描述 -> 故事 / 章节 / 微故事`

服务端只负责编排，不负责默认看图。

---

## 3. 范围

### 包含

- 默认事件故事生成重构
- 章节故事生成重构
- 微故事/caption 生成重构
- 后端 schema 与 prompt 重构

### 不包含

- Android 原生识别实现
- 默认导入链路重构
- 增强入口上传与增强故事生成

---

## 4. 产品要求

### 4.1 默认故事输入

默认故事生成必须基于：

- 拍摄时间范围
- GPS / 地理编码
- 事件地点上下文
- 照片结构化识别摘要
- 事件时间顺序

### 4.2 默认故事输出

仍需支持：

- 事件标题
- 事件摘要
- 完整故事
- 章节文案
- 微故事 / 字幕

### 4.3 质量目标

- 虽然默认故事质量允许弱于增强版
- 但应保证时间、地点、活动与情绪有基本连贯性

---

## 5. 技术方向

### 后端逻辑

- 删除默认链路中“公开图片 URL 收集与视觉分析”前提
- 新增结构化描述聚合逻辑
- prompt 从“按图叙事”改为“按结构化旅行线索叙事”
- 保留增强入口专用的云端看图分支，不与默认链路混用

### 数据侧

- 事件级需要有结构化摘要聚合
- 照片级结构化结果需可用于章节切分与封面文案

## 7. 验收口径

- 默认故事生成不再依赖图片 URL
- 在默认不上图前提下，事件仍可生成完整故事与章节
- 失败原因与任务状态仍可追踪
- 增强链路与默认链路在代码层清晰分开

---

# 需求文档：自动导入、自动事件聚合与故事生成改造

- 文档状态：Draft
- 优先级：P0
- 负责人：AI 协作
- 创建时间：2026-04-01
- 最后更新：2026-04-01

## 1. 背景

- 当前现状：
  - 默认链路虽然已经转向“metadata + 端侧结构化结果”，但首次入口仍然是用户手动选图
  - Android 相册多选体验差，用户很难一次高效选择大量照片
  - 当前产品更像“手动上传素材”，而不是“自动整理旅行回忆”
  - 自动时间聚合的产品价值没有成为主路径，用户需要承担过多整理动作
- 为什么要改：
  - 产品目标是单设备、隐私优先、本地化，而不是让用户先学会管理素材
  - 只上传 metadata 与端侧结果的前提下，首次自动导入并不会破坏隐私承诺
  - 自动导入最近少量照片可以降低首次服务器压力，也有利于建立更安全的用户心智
- 触发这次变更的原因：
  - 现有“手动上传/手动选图”方案不合理
  - 时间聚合没有成为稳定主链路
  - 需要重新定义“导入 -> 聚合 -> 端侧识别 -> 云端故事 -> 端侧幻灯片”的顺序

## 2. 目标

- 本次要解决什么问题：
  - 把首次体验从“手动选图导入”改成“默认导入最近 200 张”
  - 把事件生成重新定义为系统自动聚合，而不是用户先选图形成事件
  - 把故事生成触发条件改成“事件边界确定且照片结构化结果准备完成”
- 预期结果是什么：
  - 用户授权后即可快速得到事件列表和基础故事
  - 默认链路继续坚持不上图，只上传 metadata 与端侧结构化结果
  - 用户只在自动聚合不准确时做修正
- 成功标准是什么：
  - 首次导入可在不手动多选照片的前提下完成
  - 去重后的新增照片可自动归类到事件
  - 事件准备完成后可自动得到故事结果

## 3. 非目标

- 这次明确不做什么：
  - 不做跨设备同步
  - 不恢复默认上传缩略图或原图
  - 不要求用户逐个确认事件后再开始默认故事生成
  - 不把“手动选图”继续保留为首次主入口
- 哪些相关问题先不处理：
  - 不在 P0 做专门的事件合并/拆分按钮
  - 不在 P0 把端侧识别升级为本地长文本生成
  - 不在这次需求中重构完整增强上传方案细节

## 4. 用户场景

- 用户是谁：
  - 有较多旅行照片、希望快速整理回看的普通用户
  - 对照片隐私敏感，不接受默认上传原图的用户
- 典型使用场景：
  - 用户首次打开应用并授权相册
  - 系统自动导入最近 200 张旅行照片
  - 系统自动生成多个事件，随后自动补齐故事
  - 用户对聚合不准的事件进行修正
  - 用户对重要事件主动上传少量代表图做增强
- 关键操作路径：
  - 首次授权 -> 自动导入最近 200 张 -> 去重 -> 自动聚合事件
  - 端侧后台识别照片内容 -> 事件就绪 -> 云端生成故事
  - 用户进入事件详情 -> 修正照片归属或编辑事件 -> 重新生成故事

## 5. 需求拆解

### 5.1 功能需求

- 需求 1：首次授权后默认导入最近 `200` 张照片
- 需求 2：导入时先基于 metadata 去重
- 需求 3：去重后的新增照片自动参与时间/GPS 聚合并形成事件
- 需求 4：端侧按照片提取结构化内容，并与照片记录绑定
- 需求 5：当某个事件的照片内容准备完成后，自动触发云端故事生成
- 需求 6：允许用户再次导入新增照片
- 需求 7：保留手动补导入能力，但降级为次级入口
- 需求 8：支持事件增删改查，以及事件内增加/减少照片
- 需求 9：保留事件级代表图上传增强入口，且只能由用户主动触发

### 5.2 交互 / 界面要求

- 页面或入口：
  - 首次进入事件页或导入页时，主动作应是“导入最近 200 张”
  - “手动补导入”作为次级入口保留
- 用户操作：
  - 用户授权相册后，无需手动多选即可启动首次整理
  - 用户可在事件详情中修正事件和照片归属
  - 用户可主动进入增强入口上传少量代表图
- 状态反馈：
  - 明确展示导入数量、去重数量、失败数量
  - 明确展示“正在聚合事件”“正在分析照片内容”“正在生成故事”等状态
  - 明确展示默认不上图，只上传 metadata 与端侧结构化结果
- 异常提示：
  - 相册权限拒绝时提示去系统设置开启
  - 导入失败时提示可重试
  - 故事生成失败时提示该事件可重新生成

### 5.3 数据 / 状态要求

- 需要新增或调整的数据：
  - 首次自动导入批次状态
  - 已导入资产引用列表
  - 端侧结构化结果本地缓存
  - 事件是否待重新生成的状态
- 状态流转：
  - 授权中 -> 扫描中 -> 去重中 -> 聚合中 -> 端侧识别中 -> 故事生成中 -> 完成
  - 事件编辑后 -> 待重新生成 -> 生成中 -> 完成 / 失败
- 持久化要求：
  - 本地持久化照片引用和端侧识别缓存
  - 云端持久化 metadata、结构化结果、事件关系和故事文本
  - 默认不持久化原图或缩略图

## 7. 方案约束

- 必须遵守的业务约束：
  - 首次默认导入数量固定为 `200`
  - 默认链路不得上传原图或缩略图
  - 自动聚合是主路径，人工修正是补救手段
  - 云端增强入口必须由用户主动触发
- 性能约束：
  - 首次导入不能因端侧识别而长时间阻塞主流程
  - 故事生成应按事件分批触发，避免全量等待
- 安全 / 隐私约束：
  - 默认链路只上传 metadata 与端侧结构化结果
  - 增强入口上传内容需明确告知用途和范围
- 兼容性约束：
  - 不以历史兼容为优先目标
  - 违背当前产品方向的旧链路可直接删改

## 8. 风险与待确认

- 风险 1：
  - 自动聚合质量不足时，事件编辑能力必须足够可用，否则用户会失去信任
- 风险 2：
  - 端侧识别完成速度不稳定时，故事生成可能延迟较久
- 还需要确认的问题：
  - 后续“再次导入新增照片”是基于最近时间窗口还是基于已导入资产差集
  - 事件编辑后是否立即自动重生成，还是允许用户手动触发重生成

- [ ] 首次进入时，用户可以直接启动“导入最近 200 张”，不再依赖手动多选

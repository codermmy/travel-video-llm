# 需求文档：导入后异步端侧识别与渐进式故事生成

- 文档状态：Draft
- 优先级：P0
- 负责人：AI 协作
- 创建时间：2026-04-01
- 最后更新：2026-04-01

## 1. 背景

- 当前现状：
  - 现在默认链路虽然已经改成“最近 200 张 -> metadata 去重 -> 聚合事件 -> 端侧结构化结果 -> 云端故事”
  - 但实际执行方式仍然更接近“导入时同步等待端侧分析完成，再上传 metadata”
  - 导入主流程会被端侧识别拖长，用户需要等待较久，且在等待阶段无法自然浏览和修正事件
  - 这和“旅行回忆自动整理”的产品心智不一致，更像一次耗时上传任务
- 为什么要改：
  - 导入完成后，用户应该尽快看到事件和基础结果，而不是被端侧分析阻塞
  - 端侧识别本质上是补齐结构化描述的后台整理工作，适合在应用进程仍存活时持续推进
  - 故事生成也不应强依赖全量照片一次性就绪，而应按事件准备度逐步推进
- 触发这次变更的原因：
  - 已经确认 Android 端侧模型可用，结构化字段已能稳定产出
  - 当前同步等待方式体验偏重，不符合产品希望的“先可用，再持续补齐”方向

## 2. 目标

- 本次要解决什么问题：
  - 把端侧识别从“导入主流程同步等待”改成“导入完成后应用内异步后台队列”
  - 让用户在导入和事件聚合完成后即可进入足迹和事件页继续操作
  - 让故事生成按事件就绪度渐进式触发，而不是等待整批照片全部识别完成
  - 当应用仍在运行但切到后台时，明确提示“仍在整理中，保持应用开启可加快完成”
- 预期结果是什么：
  - 用户授权相册后，较快得到可浏览的事件列表
  - 端侧识别在应用前后台切换过程中继续推进，只要进程还活着就不中断
  - 事件的故事会逐个补齐，而不是整批卡住
- 成功标准是什么：
  - 导入流程不再因端侧识别显著阻塞首屏可用性
  - 用户在识别进行中可以正常浏览、查看事件详情、修正照片归属
  - 识别和故事生成状态可追踪、可解释、可恢复

## 3. 非目标

- 这次明确不做什么：
  - 不在 P0 承诺用户杀死应用后仍稳定继续跑端侧识别
  - 不在 P0 引入 Android 前台服务、WorkManager 或系统级持久后台任务
  - 不在这次需求中重做 iOS 端侧识别能力
  - 不在 P0 改成端侧生成长文本故事
- 哪些相关问题先不处理：
  - 不在 P0 做复杂的队列优先级和调度策略
  - 不在 P0 做照片级“暂停 / 恢复 / 重试”手动管理界面
  - 不在 P0 做完整的截图过滤或素材质量筛选规则升级

## 4. 用户场景

- 用户是谁：
  - 想尽快看到旅行事件结果、不愿长时间等待导入完成的普通用户
  - 对隐私敏感，希望默认仍不上图的用户
- 典型使用场景：
  - 用户首次授权相册，系统导入最近 200 张
  - 系统很快完成 metadata 去重和事件聚合，用户已能进入事件列表
  - 端侧识别在应用仍运行时异步持续推进
  - 某个事件的照片结构化结果已就绪，系统自动触发该事件故事生成
  - 用户在识别未完成时查看事件、修正照片归属或切回足迹页
  - 用户把应用切到后台，系统提示“仍在整理中，建议暂时不要关闭应用”
- 关键操作路径：
  - 首次授权 -> 自动导入最近 200 张 -> metadata 去重 -> 自动聚合事件 -> 事件列表可浏览
  - 端侧异步识别持续运行 -> 事件逐个达到可生成条件 -> 云端按事件生成故事
  - 用户修正事件 -> 该事件重新进入待生成 -> 故事重新生成

## 5. 需求拆解

### 5.1 功能需求

- 需求 1：导入主流程完成 metadata 上传和事件聚合后，不再等待整批端侧识别完成
- 需求 2：端侧识别改为应用内异步后台队列，只要应用进程仍存活就继续推进
- 需求 3：照片需有明确的端侧识别状态，例如 `pending / processing / completed / failed / unsupported`
- 需求 4：事件需有明确的“结构化结果准备度”状态，用于决定是否可以自动触发故事生成
- 需求 5：故事生成改为按事件粒度渐进式触发，不要求整批照片全部识别完
- 需求 6：当用户修改照片归属或事件内容后，仅让受影响事件重新进入待生成状态
- 需求 7：当应用切到后台且仍有识别任务时，提示用户保持应用开启可加快完成
- 需求 8：当用户尝试离开应用时，若仍有较多任务未完成，可给予提醒，但不做强阻断
- 需求 9：识别失败或未完成的照片可在后续应用存活期间继续补跑或重试

### 5.2 交互 / 界面要求

- 页面或入口：
  - 足迹页 / 事件页在导入完成后即可展示已聚合出的结果
  - 不再把“端侧识别中”作为阻塞整个导入弹窗关闭的前置条件
- 用户操作：
  - 用户可在识别过程中继续浏览事件列表、进入详情、查看足迹
  - 用户切到后台时，如仍在整理，可看到明确提示
  - 用户回到应用时，可继续看到当前整理进度
- 状态反馈：
  - 明确展示“正在聚合事件”“正在后台分析照片内容”“正在生成故事”
  - 事件级应能区分“已导入但待分析”“分析中”“故事待生成”“故事生成中”“已完成”
  - 提示语需明确：默认仍不上图，只上传 metadata 与端侧结构化结果
- 异常提示：
  - 当应用被杀掉或进程中断导致分析未完成时，重新进入应用后应能提示“继续整理”
  - 当设备性能或原生模块不可用时，应清楚提示“端侧识别不可用，部分故事会延后”

### 5.3 数据 / 状态要求

- 需要新增或调整的数据：
  - 导入批次状态与是否已完成主流程
  - 照片级端侧识别状态
  - 事件级结构化准备度
  - 事件待生成 / 生成中 / 完成 / 失败状态
  - 当前队列是否仍有任务待处理
- 状态流转：
  - 导入批次：授权中 -> 扫描中 -> 去重中 -> 聚合中 -> 可浏览
  - 照片识别：pending -> processing -> completed / failed / unsupported
  - 事件故事：waiting_for_vision -> ai_pending -> ai_processing -> generated / ai_failed
  - 应用切后台：processing -> background_hint_shown -> 继续处理 / 中断待恢复
- 持久化要求：
  - 本地持久化端侧识别缓存与队列必要状态
  - 云端持久化 metadata、结构化结果、事件关系和故事文本
  - 默认不持久化原图或缩略图

## 7. 方案约束

- 必须遵守的业务约束：
  - 默认链路仍只上传 metadata 与端侧结构化结果
  - 事件一旦可浏览，就不应再被整批识别任务锁死
  - 自动故事生成应按事件逐步推进，而不是整批等待
- 性能约束：
  - 导入主流程需优先保证首屏可用性
  - 识别任务不能明显阻塞 UI 滑动、浏览和详情进入
  - 事件故事生成应按事件分批触发，避免大批量串行等待
- 安全 / 隐私约束：
  - 不因异步化而改变默认不上图原则
  - 后台提示不得暗示会上传原图
- 兼容性约束：
  - P0 仅保证“应用进程仍存活时”的异步后台识别
  - 若后续需要离开应用持续处理，需单独立项做原生后台方案

## 8. 风险与待确认

- 风险 1：
  - 事件先展示、故事后补齐后，用户可能看到“空故事 / 半成品事件”，需要状态提示足够清楚
- 风险 2：
  - 应用进入后台后被系统回收，识别任务会中断，需要恢复策略和文案提示
- 风险 3：
  - 如果按事件逐步触发故事生成，可能出现同一导入批次里故事陆续补齐、用户感知不一致
- 还需要确认的问题：
  - 事件故事是要求“事件全部照片识别完成后再生成”，还是达到某个最小覆盖率即可先生成
  - 前台与后台状态提示是放在全局横幅、任务弹窗，还是事件列表顶部状态条

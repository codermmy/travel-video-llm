# 移动端自治优化循环上下文

这份上下文给 `scripts/ui_loop_runner.py` 使用，目的是让 agent 在 12 小时自治迭代时始终围绕当前产品方向收口，而不是越跑越散。

## 当前目标

- 持续把 `mobile/` 里的 Expo 移动端 UI 拉向 `2026-04` 的返工基线。
- 重点不是做“更花的 UI”，而是：
  - 更真实的任务状态
  - 更直接的用户任务路径
  - 更清晰的信息架构
  - 更强的旅行记忆气质
  - 更克制的信息密度
  - 更稳定的视觉层级

## 当前产品方向

- 当前项目没有历史兼容包袱，不以旧心智为优先目标。
- “手动选图是主入口”不再成立，只保留为补充导入能力。
- 当前口径是“旅行记忆整理产品”，不是“功能堆叠工具箱”。
- 如果旧结构阻碍当前方向，可以直接重构或删除。

## 移动端返工基线

- 页面结构尽量按 HTML 设计稿 1:1 对齐。
- 首页、地图、任务中心的状态必须基于真实任务态，不允许继续猜测“整理中 / 分析中”。
- Tab 壳层只保留 `回忆 / 地图 / 我的`。
- 事件详情首屏要做减法，只保留必要信息和主动作。
- 页面优先级上，首页、地图、事件详情、照片链路、导入链路是核心。

## 当前已确认的问题倾向

- 首页、地图、详情页仍容易残留“实现化”痕迹，而不是“设计稿心智”。
- 多处页面虽然功能存在，但层级、顺序、节奏、状态语言仍不够收口。
- 回忆流、详情页、地图层都容易出现信息偏重、说明文案偏多、状态表达不够真实的问题。
- 视觉目标不是多加装饰，而是做减法、做层级、做节奏、做可信赖感。

## 必须优先检查的维度

- 产品目标是否在首屏清楚传达
- 用户最重要的任务是否足够直接
- CTA 是否清晰
- 状态是否真实
- 空状态 / 错误状态 / 加载状态是否完整
- 信息密度是否过重
- 卡片、胶囊、抽屉、浮层是否统一
- 移动端是否符合单手使用和低认知负担
- 文案是否具体，而不是空泛描述
- 是否存在“看起来更精致但更难用”的伪优化

## 明确不要做的事

- 不新增后端接口
- 不借机扩展新业务能力
- 不把范围扩到分享、导出、搜索等低频页
- 不为了统一而把真实任务态改回假状态
- 不用大面积样式翻新掩盖结构问题
- 不在单轮里改太多无关模块

## 推荐本轮重点文件

- `src/screens/memories-screen.tsx`
- `app/(tabs)/index.tsx`
- `app/(tabs)/map.tsx`
- `app/events/[eventId].tsx`
- `app/(tabs)/profile.tsx`
- `src/components/ui/revamp.tsx`
- `src/styles/theme.ts`
- `src/styles/colors.ts`

## 自治循环原则

- 每轮先审查，再自我挑战，再执行，再校验，再严苛复审。
- 每轮只抓 1 到 3 个高杠杆问题。
- 如果已经足够好，就换下一个瓶颈，不反复抠同一点。
- 只要还有明显高优先级问题，就继续下一轮。
- 如果连续几轮没有实质提升，就停止并输出当前瓶颈。
